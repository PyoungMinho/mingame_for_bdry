## 0. 설계 전제 요약 (방향서 계승)

- **모바일 우선**: 블라인드/카톡에서 대부분 모바일로 유입 → 375px 기준 설계, 최대 480px 컨테이너.
- **순수 클라이언트 로직**: DB/API 의존 0. 모든 판정은 브라우저에서 즉시 계산.
- **테스트 엔진 재사용**: 질문·유형 데이터는 `src/app/office-archetype/data/*.json`으로 완전 분리. UX 상에서도 "화면 로직"과 "콘텐츠"가 분리되어 있다는 전제로 설계(향후 hsp-test, dating-balance 등이 같은 화면 골격 재사용).
- **정확도 리스크가 최우선 게이트**: "이거 내 얘기 아닌데?" 소리가 나오면 부정 바이럴로 역전 → 카피 톤·문항 설계·베타 검증 흐름을 UX 산출물에 포함(요청 사항 5번).
- **바이럴 루프 = 상성 동료 매칭**: 1인 완결형 결과를 2차 공유로 강제 전환하는 장치가 핵심 KPI(K-factor ≥ 1).

## 1. 사용자 페르소나

### 페르소나 A — 핵심: "점심시간 3분" 유정 (31세, 대기업 대리)
- **상황**: 점심 먹고 자리에 앉아 스마트폰으로 블라인드/단톡방을 훑는다. 동료가 공유한 링크를 탭.
- **목표**: 킬링타임 + "내가 어떤 사람인지" 재확인하는 재미. 결과가 그럴듯하면 바로 부서 단톡에 공유.
- **불안**: "또 그런 뻔한 MBTI 테스트겠지" (피로도 높음) / "내 얘기 아니면 스킵".
- **행동 패턴**: 모바일 100%, 세로 스크롤 최소 선호, 텍스트 길면 안 읽고 이탈. 결과 공유는 캡처보다 원클릭 공유 버튼 선호.
- **결제 심리**: 무료 기대. 광고 배너는 허용하나 강제 시청형은 즉시 이탈.

### 페르소나 B — 확산 매개: "우리 팀 궁합 챙기는" донор형 재현 (34세, 팀장급)
- **상황**: 본인 테스트 후 결과의 "상성 좋은 동료 유형"을 보고 실제 팀원을 떠올림.
- **목표**: "너 이 유형 아니야?" 하며 동료에게 링크를 재전송 — 자신이 유형 매칭의 발신자가 되고 싶어함.
- **불안**: 결과가 너무 뻔하면(예: MBTI 재탕) 공유 안 함. "이건 좀 신선한데?" 라는 느낌이 있어야 공유.
- **행동 패턴**: 결과 카드를 부서 단톡/친한 동료 개인 카톡으로 지목 공유("@민호 이거 완전 너").

### 페르소나 C — 회의적 관찰자: "테스트 불신러" 승우 (28세, 스타트업 주니어)
- **상황**: 이런 심리테스트에 냉소적이지만 동료가 재차 언급하면 결국 눌러본다.
- **목표**: 결과가 허접하면 "역시 뻔함" 이라며 조롱 댓글/캡처로 부정 확산. 결과가 정교하면 오히려 적극 옹호자로 전환.
- **UX 함의**: 이 페르소나가 "그럴듯하다"고 느끼게 하는 카피 디테일이 정확도 리스크 대응의 실제 타겟.

## 2. 사용자 플로우

### 2-1. 전체 플로우 (4화면 + 분기)

```
[랜딩/인트로]  /office-archetype
      │  "시작하기" 탭 (3초 이내 "이게 뭔지" 이해되어야 함)
      ▼
[질문 1/10]  /office-archetype/test?q=1
      │  선택지 탭 → 자동 다음 문항 (별도 "다음" 버튼 없음, 마찰 제거)
      │  ⟲ 뒤로가기 가능 (답변 유지)
      ▼
[질문 2~10]  … 진행률 바 10칸
      │
      ▼ (10문항 완료 시 자동)
[판정 로딩]  0.6~1.0초 연출 (실제 계산은 즉시, 체감 신뢰감 위한 최소 연출)
      │
      ▼
[결과]  /office-archetype/result/[type]  (예: /result/hunter)
      │  8유형 중 1개 + 상성 동료 매칭 + 공유 CTA
      │
      ├─→ [공유 카드 생성/공유] → 카톡/인스타 원클릭
      │        │
      │        └─→ (수신자가 링크 클릭) → 랜딩으로 재유입 (바이럴 루프 완성)
      │
      └─→ [다시하기] → 질문 1로 리셋 (답변 초기화 확인 없이 즉시, 마찰 최소화)
```

### 2-2. 이탈 방지 장치 (단계별)

| 단계 | 이탈 리스크 | 방지 장치 |
|---|---|---|
| 랜딩 | "뭔지 모르겠다" → 즉시 이탈 | **3초 이해 원칙**: 헤드카피 한 줄 + 소요시간("1분") + 결과 예고 썸네일(8유형 아이콘 그리드 흐림 처리로 "궁금증" 유발) |
| 랜딩 | "느리게 로드된다" | 이미지 0~1장, 폰트는 시스템 폰트 우선. 순수 클라이언트라 API 대기 없음 → 즉시 인터랙티브 |
| 질문 1~3 | "몇 개나 남았지?" 불안 | 진행률 바 상단 고정 노출("3/10") + 첫 문항부터 진행률 보임 |
| 질문 중반(4~7) | 집중력 저하, 지루함 | 문항 텍스트 1줄 원칙(길어도 2줄), 선택지 2~4개, 탭 즉시 다음 문항 전환(애니메이션 150ms, 대기 없음) |
| 질문 후반(8~10) | "다 왔는데 관두기 아까움" 심리를 활용 — sunk cost | 진행률 바가 90% 근접 시 은은한 강조(색 진해짐)로 "거의 다 왔다" 신호 |
| 질문 전체 | 실수로 오답 탭 | **뒤로가기 항상 노출**(헤더 좌측), 이전 답변 유지한 채 재선택 가능 |
| 결과 로딩 | "버벅이나?" 오해 | 0.6~1.0초 짧은 판정 연출(스켈레톤 대신 "당신의 유형을 분석 중…" 텍스트 + 인디케이터) — 없으면 오히려 "이거 그냥 랜덤 아니야?" 불신 유발하므로 의도적으로 넣는 지연 |
| 결과 | 광고/CTA 과다로 신뢰 저하 | 결과 화면 최상단은 반드시 유형 결과(광고 없음). 광고는 결과 카드 하단, 공유 CTA보다 아래 배치 |
| 결과→공유 | "공유하기 귀찮음"(탭 3번 이상이면 이탈) | **1탭 공유**: 카카오톡 공유하기 SDK/Web Share API 직결, 별도 로그인/이미지 저장 단계 없음 |
| 공유 카드 생성 | 이미지 생성 대기 시간 | 결과 진입 시점에 카드 이미지 **미리 렌더링**(백그라운드), 공유 탭 시점엔 이미 준비된 상태로 즉시 공유 시트 오픈 |

### 2-3. 3초 이해 원칙 적용 지점

- **랜딩 히어로**: "당신의 직장 유형은? 10문항, 1분" — 제목+부제 2줄 이내, 스크롤 없이 뷰포트 최상단에 위치.
- **질문 화면 진입 즉시**: 헤더에 "1/10" 진행률 배지 — 몇 걸음 남았는지 즉시 인지.
- **결과 화면 최상단**: 유형 이름 + 한 줄 캐치프레이즈(예: "회의실의 조용한 해결사") — 스크롤 없이 3초 내 "내 유형이 뭔지" 파악 가능. 상세 설명은 그 아래.

## 3. 정보 구조 (IA)

### 3-1. 화면 계층 및 라우팅

```
/office-archetype                          랜딩/인트로 (엔트리 포인트)
/office-archetype/test                     질문 진행 (내부 state로 q1~q10 관리, URL은 고정 — 새로고침 시 진행 유지 위해 localStorage)
/office-archetype/result                   결과 (직접 진입 시 미완료 상태 → 랜딩으로 리다이렉트)
/office-archetype/result/[typeSlug]        딥링크 결과 공유용 (예: /result/hunter, /result/builder)
                                            → 공유받은 사람이 접속 시 "친구의 유형" 프리뷰 화면으로 표시
                                            → "나도 해보기" CTA로 /office-archetype 랜딩 이동
```

> 비고: `/test`를 URL 쿼리(`?q=1`)로 문항 번호를 노출할지, 순수 내부 state로 관리할지는 개발팀 판단 영역이나, **뒤로가기 시 이전 문항으로 돌아가야 하는 요구사항**(요청 2번) 때문에 브라우저 히스토리와 문항 state를 동기화하는 방식(`history.pushState` 또는 Next router shallow) 권장. 최소 구현은 컴포넌트 내부 state + `popstate` 리스너로도 충분.

### 3-2. 딥링크 결과 공유 설계 (바이럴 핵심 IA)

- 결과 확정 시 판정 로직은 클라이언트에서 실행되지만, **결과 유형은 URL에 인코딩**되어 정적 공유가 가능해야 한다.
- `/office-archetype/result/[typeSlug]`는 정적 라우트(8유형 = 8개 슬러그, `generateStaticParams`로 사전 생성 가능) → OG 메타 태그를 유형별로 서버에서 정확히 내려줄 수 있음(카톡 프리뷰 정확도).
- 딥링크 접속자에게는 자신이 테스트를 안 한 상태이므로 **"친구 결과 프리뷰" 모드**로 구분 표시:
  - 상단 배지: "민호님이 공유한 결과예요" (리퍼러 정보 없으면 "누군가 공유한 결과예요")
  - 본문: 해당 유형 설명 전체 노출(호기심 자극)
  - 하단 고정 CTA: "나는 무슨 유형일까? 1분 테스트 시작" → `/office-archetype`
- 이 구조로 **1회 공유 → N회 신규 유입**의 바이럴 루프가 URL 레벨에서 성립.

### 3-3. 상태(State) 목록

| 상태 | 저장 위치 | 용도 |
|---|---|---|
| 현재 문항 인덱스 (0~9) | React state + `sessionStorage` | 새로고침/뒤로가기 복원 |
| 각 문항 답변 배열 | `sessionStorage` (`office-archetype-answers`) | 뒤로가기 시 이전 선택 유지, 결과 계산 입력값 |
| 진행률 | 파생값 (answers.length / 10) | UI 표시용, 별도 저장 불필요 |
| 결과 유형 | 계산 후 `sessionStorage` + URL(`/result/[slug]`) | 새로고침 시 재계산 없이 즉시 표시 |
| 재시작 플래그 | 즉시 `sessionStorage.clear()` 후 리다이렉트 | "다시하기" 시 확인 모달 없이 즉시 초기화(마찰 제거 우선, 되돌릴 데이터 가치 낮음) |
| 공유 카드 렌더 캐시 | 메모리(state) 또는 Canvas 오프스크린 | 결과 진입 시 미리 생성해 공유 탭 지연 제거 |

### 3-4. 네비게이션 맵

```
랜딩 ──"시작하기"──▶ 질문(1) ──자동진행──▶ … ──▶ 질문(10) ──자동──▶ 판정 로딩 ──▶ 결과
  ▲                    │← 뒤로가기(헤더)                                    │
  │                    ▼                                                    │
  │                  랜딩 (질문 1에서 뒤로가기 시)                            │
  │                                                                         │
  └──────────────────────── "다시하기" ────────────────────────────────────┘
  ▲
  │  "나도 해보기" CTA
  │
딥링크 결과(/result/[type]) ← 외부 공유 링크로 진입(테스트 미완료자)
```

## 4. 와이어프레임 (4화면)

공통 레이아웃 규칙: `max-w-[480px] mx-auto`, 상단 고정 헤더 `h-14`, 하단 CTA는 `StickyBottomBar` 패턴(moira 관례 계승), safe-area 패딩 적용.

### 화면 1 — 랜딩/인트로 (`/office-archetype`)

```
┌─────────────────────────────────────┐
│                                     │  (헤더 없음 또는 최소 워드마크만, 몰입 우선)
│         [8유형 아이콘 그리드]        │  ← 흐릿하게(blur) 처리된 8칸 그리드
│         ⬡⬡⬡⬡                       │    "내 유형은 이 중 뭘까?" 궁금증 유발
│         ⬡⬡⬡⬡                       │
│                                     │
│   당신의 직장 유형은?                │  ← H1, 28px, 2줄 이내
│   10문항 · 1분 · 결과 100% 무료      │  ← 소요시간·비용 명시 (기대치 관리 = 이탈 방지)
│                                     │
│   "MBTI 말고, 진짜 회사에서의       │  ← 서브카피: 차별점 명시
│    내 모습을 알려드려요"             │
│                                     │
│  ┌───────────────────────────────┐  │
│  │        시작하기  →            │  │  ← 1차 CTA, 풀폭 버튼, h-14
│  └───────────────────────────────┘  │
│                                     │
│   지금까지 12,483명 참여 (예시)      │  ← 사회적 증거(카운터, 클라이언트 localStorage 누적치 또는 정적 숫자)
│                                     │
│  ─────────────────────────────────  │
│  [광고 슬롯 — 결과보다 아래 원칙과   │  ← 랜딩에도 넣는다면 CTA 하단, 스크롤 필요 영역에만
│   동일하게 CTA 아래 배치]            │
└─────────────────────────────────────┘
```

**핵심 요소**
- H1 카피는 반드시 "직장/회사" 맥락 명시 (일반 심리테스트와 차별화, 정확도 리스크 대응의 첫 인상 관리).
- "MBTI 말고" 문구로 기획 방향서 §5 "MBTI 재탕 금지" 원칙을 사용자에게도 각인 → 기대치를 "새로운 것"으로 설정해 정확도 판단 기준 자체를 낮춤(관대하게 받아들이게 하는 프레이밍).
- 참여자 수 카운터는 초기엔 정적 텍스트("첫 주 오픈!")로 시작해도 무방, 이후 실제 카운트 붙이면 사회적 증거 강화.

### 화면 2 — 질문 (`/office-archetype/test`)

```
┌─────────────────────────────────────┐
│  [←]        ▓▓▓▓▓▓░░░░  4/10        │  ← 헤더 h-14, 뒤로가기 + 진행률 바(10칸 세그먼트)
├─────────────────────────────────────┤
```

**핵심 요소 — "한 손 조작" 설계**
- 선택지는 화면 하단 60% 영역(엄지 도달 구간)에 오도록 문항 텍스트를 상단에 압축 배치.

- 진행률 바는 10칸 세그먼트 방식(moira의 연속 바와 차별화, "몇 개 남았는지" 개수 감각을 더 직관적으로 전달).

- 문항 1에서 뒤로가기 시 랜딩으로 이동(확인 모달 없음).

## 5. 바이럴 확산 루프 설계 (K-factor ≥ 1 지향)

### 5-1. 루프 구조 — "상성 동료 매칭"의 2차 공유 전환

```
[1차 공유] 사용자 A가 결과를 자기 계정/단톡에 공유
    │
    │  카드에 "상성 좋은 동료: 에너자이저형" 노출
    ▼
[지목 트리거] A가 실제 동료 B를 떠올리며 "이거 완전 너 아니야?" 개인 카톡으로 재전송
    │  ★ 이 지점이 팀 단위 확산의 핵심 — 방향서 §4-2 "1인 완결형 → 2~n인 참여형 전환"
    ▼
[2차 유입] B가 링크 클릭 → 딥링크 결과 프리뷰(/result/[type]) 확인
    │  "친구가 공유한 결과" 화면에서 A의 유형 설명 확인
    ▼
[전환 CTA] "나도 무슨 유형인지 궁금하다" → "1분 테스트 시작" 버튼
    │
    ▼
[B가 테스트 완료] → B의 결과에도 "상성 좋은 동료" 매칭 노출
    │
    └─→ B가 "어 근데 A랑 나랑 상성 좋다는데?" 하며 A에게 재공유 → 왕복 공유 루프 발생
```

### 5-2. K-factor를 끌어올리는 구체 장치

| 장치 | 설계 내용 |
|---|---|
| **지목형 카피** | 결과 화면에 "이 유형, 당신 주변에 누가 있나요?" 프롬프트 + 카톡 공유 시 기본 텍스트에 "@[친구 이름 넣기]" 형태 placeholder 유도 (Web Share API의 text 파라미터에 "이거 완전 너 아니야? →" 프리셋) |
| **상호 매칭 확인 욕구** | 상성 동료 카드에 "[이 유형 보기 →]" 링크 배치 → 탭 시 해당 상성 유형의 결과 화면(자기 테스트 없이 유형 설명만)을 프리뷰로 보여줌 → "나도 이거일까?" 궁금증 → 테스트 유도 |
| **팀 단위 프레이밍** | 방향서 §7 "우리 팀 전체 유형 리포트" 옵션과 연결 — 결과 화면 하단에 "우리 팀은 총 몇 명이 몇 유형일까?" 티저 문구(향후 소액 유료 기능 훅, 초기엔 카피만 노출해 수요 검증) |
| **공유 마찰 제거** | 공유 버튼 1탭 = 카카오톡 공유 시트 직행(로그인/이미지 저장 단계 없음). Web Share API 우선 사용, 미지원 시 카카오 SDK(`Kakao.Share.sendDefault`) 폴백, 최후 폴백은 클립보드 복사 + 토스트 |
| **결과 URL 영속성** | `/result/[typeSlug]`가 정적이라 몇 달 뒤 공유돼도 동일하게 작동 — 링크 수명 = 무한(방향서 §4-3 "포트폴리오 지속화"와 궤 같이함) |
| **되돌아오는 링크** | 딥링크 프리뷰 화면 CTA 옆에 작게 "이 유형 만든 사람 결과도 보기" 문구는 넣지 않음(개인정보 노출 리스크) — 대신 "나도 테스트하기"만 강조해 순환을 단순하게 유지 |

### 5-3. 측정 가능하게 만들기 (UX 관점에서 필요한 최소 계측 지점)

DB 없이도 클라이언트 이벤트로 측정 가능해야 K-factor를 실제로 검증할 수 있음. 개발팀에 요청할 최소 이벤트:
- `test_start`, `test_complete`(유형별), `share_click`(채널별: kakao/insta/copy), `deeplink_view`(유입 유형별), `deeplink_to_test_start`(전환) — Vercel Analytics 또는 GA4 이벤트로 충분, 별도 백엔드 불필요.

## 6. 정확도 리스크 대응 UX (⚠️ 방향서 §5 최우선 게이트)

### 6-1. 카피 톤 가이드 — "내 얘기 맞다" 공감을 만드는 문장 원칙

| 원칙 | 설명 | 예시 |
|---|---|---|
| **바넘 효과 역이용 + 구체성 결합** | 누구에게나 맞을 법한 문장("당신은 상황에 따라 다르게 행동한다")은 피하고, **직장 맥락의 아주 구체적인 장면**을 묘사해 "어? 나 저번에 딱 저랬는데" 순간을 만든다 | "MBTI는 평생 성격을 말하지만, 이건 **월요일 오전 회의에서의 당신**을 말합니다" |
| **행동 묘사 우선, 특성 라벨 나중** | "당신은 내향적입니다" 대신 "회의 끝나고 나서야 하고 싶었던 말이 떠오르는 편이죠" 처럼 행동/장면을 먼저 제시 후 라벨 부여 | 강점/그림자 문구도 형용사 나열이 아니라 장면형 문장으로 작성 |
| **긍정 프레이밍 우선 배치** | 강점을 그림자보다 먼저, 더 길게. 그림자는 "약점"이 아니라 "이럴 때 조심하면 좋아요" 톤 | "그림자" 섹션 제목 자체를 "이럴 때 주의하면 더 좋아요"로 대체 검토 |
| **회사 은어/밈 차용** | "TMI 참기", "칼퇴", "리마인드 메일", "복붙 보고서" 등 직장인이 즉각 반응하는 어휘 사용 → 공감 반응속도 향상 | 문항/설명 문구 전반에 자연스럽게 섞기(과하면 억지스러움 주의) |
| **자기방어 여지 남기기** | 그림자 문항 뒤에는 반드시 "물론 상황에 따라 다르지만" 같은 완충 문구를 넣어 반발감 최소화 | 톤앤매너 공통 규칙 |

### 6-2. 문항 설계 가이드 (콘텐츠팀/기획 인터페이스)

- **문항은 반드시 "직장 내 특정 상황"으로 시작**: 일반적 성격 질문("당신은 외향적인가요?") 금지, 대신 "팀 회식 자리 배정을 정할 때 나는?" 처럼 장면 고정.
- **선택지는 극단 2개 + 중도 1~2개** 구성 원칙(4지선다 표준) — 극단만 있으면 "나는 둘 다 아닌데" 이탈 유발.
- **8유형 축은 방향서 §5가 예시한 "일 vs 관계, 주도 vs 순응, 안정 vs 도전, 겉맴 vs 진심"** 등 직장 고유 축 사용, MBTI 4문자 재라벨링(E/I, T/F 등 그대로 차용) 금지를 UX 카피에서도 재확인(문항 진행 중 은연중 MBTI 용어 노출 금지).
- **질문 순서 배치**: 앞부분(1~3번)은 답하기 쉬운 저부담 문항(예: "점심 메뉴 정할 때")으로 시작해 이탈률 낮추고, 중후반(4~8)에 판별력 높은 핵심 문항 배치, 마지막(9~10)은 다시 가벼운 문항으로 마무리(완료 경험을 긍정적으로 남김).

### 6-3. 베타 검증 흐름 (배포 게이트, UX 프로세스로 명문화)

방향서 §5 "런칭 전 30~50명 베타 공감률 검증" 을 UX 측에서 운영 가능한 흐름으로 구체화:

```
1단계 — 콘텐츠 준비
  블라인드 인기글 200개+ 정성분석(기획팀) → 8유형 초안 + 10문항 초안(JSON) 확정

2단계 — 클로즈드 베타 UX (배포 전, 별도 페이지 불필요)
  동일한 /office-archetype 라우트를 비공개 링크로 30~50명에게 배포 (예: 아직 공유 링크 걸지 않고 개인 DM으로만 전달)
  테스트 완료 화면에 베타 전용 임시 위젯 추가:
    "이 결과, 내 얘기 같나요?" [1점 전혀 아니다 ~ 5점 정확히 내 얘기다] (5점 척도, 원탭)
    선택 시 콘솔/애널리틱스 이벤트로 (유형, 점수) 전송 — 백엔드 저장 없이 GA4 이벤트만으로 충분
    "어떤 부분이 안 맞았나요? (선택)" 짧은 텍스트 입력 1줄 — 스킵 가능

3단계 — 판정 기준 (배포 게이트)
  유형별 평균 공감 점수 4.0/5.0 미만 → 해당 유형 재설계
  전체 평균 4.0 미만 → 배포 보류, 8유형 축 재검토
  4.0 이상 유형만 정식 공개, 미달 유형은 "준비 중" 처리하거나 즉시 콘텐츠 수정 후 재검증

4단계 — 정식 공개 전환
  베타 위젯 제거(또는 상시 유지해 지속 모니터링 — 비용 0이므로 상시 유지 권장)
  베타 참여자에게 "먼저 해주셔서 감사합니다, 이제 정식 오픈했어요" 공유 유도 메시지 발송(콜드스타트 시딩 겸용)
```

> UX 설계 노트: 베타 피드백 위젯은 정식 공개 후에도 제거하지 말고 상시 유지하는 것을 권장. 비용이 이벤트 전송 1건뿐이라 "결과 정확도"를 계속 모니터링할 수 있고, 특정 유형의 공감도가 시간이 지나며 떨어지면(밈 변화 등) 콘텐츠 업데이트 신호로 활용 가능.

## 7. 접근성 · 모바일 터치 타겟 · 로딩/에러 상태

### 7-1. 접근성 (WCAG 2.1 AA)

| 항목 | 적용 |
|---|---|
| 색 대비 | 본문 텍스트 대비 4.5:1 이상. 강점(초록 계열)/그림자(주의, 앰버 계열) 구분에 색만 쓰지 않고 아이콘(💪/🌑 또는 lucide 아이콘) 병행 |
| 포커스 링 | `globals.css` 전역 규칙 계승(`focus-visible` 오렌지 3px) 또는 office-archetype 전용 브랜드 컬러로 스코프 지정(moira 사례처럼 `.office-archetype-scope :focus-visible` 오버라이드 가능) |
| 진행률 바 | `role="progressbar"` `aria-valuenow`/`aria-valuemin`/`aria-valuemax` + `aria-label="10문항 중 4번째"` |
| 선택지 버튼 | 실제 `<button>` 엘리먼트 사용(div+onClick 금지), 키보드 Tab 순서 = 시각적 순서와 일치, Enter/Space로 선택 가능 |
| 결과 화면 | 유형 이미지에 `alt="[유형명] 일러스트"`, 강점/그림자는 `<ul>` 시맨틱 리스트 |
| 공유 카드 이미지 | 다운로드/공유되는 이미지 자체는 접근성 대상 아니나, 카드 생성 실패 시 폴백 텍스트 공유(URL만) 제공 |
| 모션 감소 | `prefers-reduced-motion: reduce` 시 문항 전환 트랜지션·판정 로딩 애니메이션·진행률 바 채워짐 애니메이션 모두 즉시 최종 상태로(전역 `globals.css` 규칙 계승, 별도 예외 불필요) |
| 스크린리더 흐름 | 질문 화면 진입 시 문항 텍스트에 포커스 이동(또는 `aria-live="polite"`로 새 문항 안내) → 각 문항 전환마다 스크린리더가 새 질문을 읽도록 보장 |

### 7-2. 모바일 터치 타겟

- 모든 인터랙티브 요소 최소 44×44px (`globals.css`의 `.touch-target` 유틸 재사용).
- 질문 선택지 카드: `min-height: 56px`, 카드 간 간격 `gap: 10px` 이상 — 오탭 방지(직장인이 지하철/이동 중 사용하는 시나리오 고려).
- 하단 StickyBottomBar 버튼: `h-14`(56px) 이상, 좌우 버튼 사이 최소 12px 간격.
- 공유 버튼은 엄지 자연 도달 구간(화면 하단 25%)에 고정 배치.

### 7-3. 로딩 상태

| 시점 | 처리 |
|---|---|
| 랜딩 최초 진입 | 순수 정적 콘텐츠(클라이언트 로직 0 대기) — 로딩 상태 사실상 불필요, 이미지만 `loading="eager"`로 즉시 표시 |
| 문항 JSON 로드 | 빌드 타임에 정적 임포트(`import questions from './data/questions.json'`) 권장 → 런타임 fetch 자체가 없어 로딩 상태 불필요. fetch 방식을 택할 경우 스켈레톤 문항 카드(회색 블록 3개) 노출 |
| 판정 계산 → 결과 전환 | 의도적 0.6~1.0초 지연 연출: "당신의 유형을 분석하는 중…" 텍스트 + moira 관례의 `moira-indeterminate` 유사 바(office-archetype 전용 프리픽스로 새로 정의, 예: `oa-indeterminate`) |
| 공유 카드 이미지 생성 | 결과 진입 시점 백그라운드 프리렌더 → 사용자가 공유 버튼 누르기 전 대부분 완료. 혹시 미완료 시 공유 버튼에 짧은 스피너 오버레이(최대 1초) 후 시트 오픈 |

### 7-4. 에러 상태

| 시나리오 | 처리 |
|---|---|
| `sessionStorage` 접근 불가(시크릿 모드 등) | 진행 상태 저장 실패해도 테스트 자체는 메모리 state로 계속 진행 가능하게(저장 실패를 치명적 에러로 취급하지 않음). 단, 새로고침 시 초기화될 수 있다는 점을 문항 최초 진입 시 작은 안내 배지로 1회 고지("새로고침하면 처음부터 시작돼요") |
| 결과 화면 직접 진입(테스트 미완료, `/office-archetype/result` 접근) | 자동으로 `/office-archetype` 랜딩으로 리다이렉트 + "테스트를 먼저 진행해주세요" 토스트 |
| 딥링크 유형 슬러그가 유효하지 않음(`/result/invalid-slug`) | 404 대신 친근한 안내 화면: "존재하지 않는 유형이에요" + "테스트 시작하기" CTA(이탈시키지 않고 전환 기회로 활용) |
| 이미지/일러스트 로드 실패 | 유형별 대체 이모지(텍스트 기반)로 즉시 폴백, 레이아웃 깨짐 없이 자리 유지(`aspect-ratio` 고정) |
| Web Share API 미지원 브라우저 | 카카오 SDK 우선 시도 → 그래도 실패 시 "링크가 복사되었어요" 토스트 + 클립보드 복사로 폴백. 절대 빈 에러창/콘솔 에러 노출 금지 |
| 카카오 SDK 로드 실패(네트워크 등) | 공유 버튼 탭 시 즉시 클립보드 복사 폴백으로 전환, 사용자는 실패를 인지하지 못하고 "복사됨" 성공 경험만 하도록 |

## 8. 인터랙션 노트 (요약)

- **문항 전환**: 선택지 탭 → 150ms `translateX` 슬라이드 + 페이드, 뒤로가기 시 역방향 슬라이드로 방향성 인지 제공.
- **진행률 바**: 세그먼트 10칸, 완료된 칸은 브랜드 컬러 즉시 채움(카운트업 애니메이션 불필요, 즉시성이 더 중요).
- **결과 진입**: 유형 아이콘/일러스트 페이드인 → 유형명 → 캐치프레이즈 순으로 100ms 간격 stagger(과하지 않게, moira `mf-rise` 패턴 참고 가능하나 office-archetype 전용 키프레임으로 격리).
- **상성 동료 카드 탭**: 바텀시트 또는 별도 섹션 확장으로 해당 유형 요약 프리뷰(전체 화면 전환 없이, 컨텍스트 유지).
- **다시하기**: 확인 모달 없이 즉시 초기화 + 질문 1로 이동(마찰 최소화 우선, 방향서의 "가볍게 여러 번 참여" 철학과 일치).
- **뒤로가기(브라우저/헤더 공통)**: 질문 중엔 이전 문항, 질문 1에서는 랜딩, 결과 화면에서는 브라우저 기본 동작 허용(뒤로가기 시 질문 마지막 문항으로 돌아가지 않고 랜딩으로 — 재입력 유도 방지 위해 결과 페이지 진입 시 `history.replaceState`로 질문 히스토리 정리 권장).

## 9. 개발팀 인터페이스 노트 (엔진 재사용성 관점)

- 질문 데이터 스키마 제안:
```json
{
  "id": "q1",
  "text": "팀 회식 자리 배정을 정할 때 나는?",
  "options": [
    { "id": "a", "text": "다들 편한 대로 앉게 둔다", "scores": { "관계": 2, "순응": 1 } },
    { "id": "b", "text": "은근슬쩍 원하는 자리를 먼저 확보한다", "scores": { "주도": 2, "겉맴": 1 } }
  ]
}
```
- 유형 데이터 스키마 제안:
```json
{
  "slug": "hunter",
  "name": "회의실의 조용한 해결사",
  "alias": "사색형 해결사",
  "tagline": "말은 적지만, 결정적일 때 정확한 한마디를 던지는 사람",
  "strengths": ["논리적 판단, 감정에 안 휩쓸림", "위기 상황에서 냉정한 조언자"],
  "shadows": ["의견을 아껴서 오해를 살 수 있음", "과열된 회의에서 존재감 옅어짐"],
  "matchSlug": "energizer",
  "matchReason": "당신의 침묵을 행동으로 바꿔주는 사람"
}
```
- 이 스키마가 곧 "테스트 엔진"의 계약(contract) — 위 두 JSON만 교체하면 hsp-test, dating-balance 등 후속 시리즈가 동일 화면 컴포넌트를 그대로 재사용 가능해야 함(화면 로직에 "직장", "동료" 같은 도메인 단어를 하드코딩하지 말고 props/JSON에서 주입받도록 UI 텍스트도 데이터화 권장 — 예: "상성 좋은 동료" 라벨 자체도 유형 JSON 밖의 공통 카피 설정 파일로 분리하면 "상성 좋은 캐릭터"처럼 시리즈별 문구 교체 가능).
