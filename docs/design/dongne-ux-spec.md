## 사용자 페르소나

**"점심시간 3분 승부사" — 이대리, 32세, 수도권 사무직**
- 카톡 단톡방·오픈채팅 링크로 유입(주 진입경로). 회원가입 요구 시 즉시 이탈.
- 점심시간·출퇴근 지하철에서 한 손 모바일로 30초~2분 짧게 소비. 카톡 인앱 브라우저 사용 비중 높음(주소창 없음, 뒤로가기 불안정, `100vh` 잘림 이슈).
- "고향이 어디냐"류 지역 드립에 반응 잘함 — 우리 동네가 정답으로 나오면 캡처해서 자랑. 스포 당하는 건 싫어함(단톡방에 정답 지명 그대로 붙이는 사람에게 야유).
- 듀오링고식 "스트릭 끊기는 것"에 약함 — 습관 리텐션 후크가 통하는 유형.
- 지도 앱은 익숙하지만 "행정구역 실루엣"은 낯설다 → 첫 판은 헷갈려도 됨, 힌트(거리·방향·근접도)로 추론하는 재미가 핵심.

이 페르소나 기준으로 설계 원칙 3가지: **(1) 회원가입 없이 즉시 플레이, (2) 3분 내 결과+공유까지 완주, (3) 스포일러 없이 자랑 가능.**

## 정보 구조 (IA)

### 라우트 맵

```
/dongne                          오늘의 게임 (핵심, 단일 화면 — 결과는 인라인 확장, 별도 라우팅 없음)
/dongne/archive                  아카이브 목록 (오늘 회차 제외, 어제까지만)
/dongne/archive/[gameNo]         '어제의 동네' 해설 (SSG, gameNo < 오늘만 생성 → 미래/오늘 404)
/dongne/about                    소개
/dongne/privacy                  개인정보처리방침
/dongne/contact                  문의
/dongne/og/[gameNo]              OG 이미지 라우트 (화면 아님, 카톡 미리보기용 asset)
```

### 네비게이션 구조
- **하단 탭바 없음**(방향서 확정). 상단에 얇은 헤더 1줄만: `동네고수 · #{gameNo}` (좌) + `아카이브` 텍스트 링크(우). 그 외 크롬 최소화 — 카톡 인앱 브라우저는 화면 높이가 절대적으로 부족하므로 헤더는 44px 이내.
- `/dongne` 안에서 **게임 → 결과는 같은 페이지 내 아코디언 확장**(라우팅 X). 새로고침해도 완료 상태 그대로 복원(재방문 플로우 참조).
- `/dongne/archive/[gameNo]` 최하단에 "오늘 문제 풀러 가기 →" 링크로 항상 코어 게임 복귀 가능(허브형 IA, 되돌아오는 문이 없는 화면 금지).
- about/privacy/contact은 도달 빈도 낮음 → 헤더에 노출 안 하고 결과 화면·아카이브 하단 푸터 텍스트 링크로만 연결(화면 밀도 우선).
- 정보 계층 깊이: 홈(게임) 기준 아카이브 상세까지 **2단계**, 정적 페이지까지 **2단계** — 3탭 원칙 충족.

### 핵심 액션 1개 원칙 적용
| 화면 | 핵심 액션 |
|---|---|
| `/dongne` (진행 중) | 지명 추측 제출 |
| `/dongne` (완료 후, 인라인) | 결과 공유(복사) |
| `/dongne/archive` | 회차 선택 |
| `/dongne/archive/[gameNo]` | 오늘 게임 복귀 |

## 사용자 플로우

### A. 코어 게임 플로우 (첫 방문)

```
진입(카톡 링크) → /dongne 로딩
  → localStorage 체크: 오늘(gameNo) 플레이 기록 없음 → "신규 플레이" 분기
  → 최초 방문(온보딩 dismiss 기록 없음)이면 규칙 스트립 노출
      "누구나 6번 안에! ① 오늘의 동네 실루엣 1개 ② 6번 안에 지명 맞히기 ③ 오답마다 거리·방향·근접도 힌트"
      [시작하기] 탭 1회 → localStorage에 dismiss 기록, 이후 재노출 안 함
  → 실루엣 SVG 노출 (온디맨드 fetch: silhouettes/{today.code}.json)
  → 입력창 포커스 → 자동완성 드롭다운(입력창 "위"로 오픈, 키보드 회피)
  → 추측 지역 선택(리스트에서만 선택 가능, 자유 텍스트 제출 불가 — 행정표준코드 매칭 보장)
  → [추측하기] 탭 → 판정
      틀림 → 해당 시도 행에 거리·방향·근접도 채워짐, 다음 시도로 진행 (최대 6회)
      맞음 → 해당 시도 행 "정답! 🎯"로 표시 → 결과 카드 인라인 확장(아래 B 참조)
      6회 모두 소진 & 오답 → 실패 상태: 정답 공개 + 해설 CTA
  → (완료 후) 자정까지 카운트다운 노출
```

**입력(자동완성) 모바일 키보드 UX 세부 규칙**
1. 드롭다운은 입력창 **위쪽**에 오픈된다(방향서 R4: 인앱 키보드가 화면 하단을 가리는 문제 회피). **[디자인팀장 확정 F2·F9]** 위 오픈이 기본이되 위 가용공간<180px면 아래로 폴백. 입력창은 **sticky로 고정하지 않는다**(카톡 하단 툴바+sticky 이중 스택 사고 방지) — 포커스 시 네이티브 scroll-into-view로 키보드 위로 올라온다.
2. 입력 1글자부터 매칭(디바운스 ~150ms), 최대 6개 결과만 노출(스크롤 최소화). 이름 매칭뿐 아니라 별칭(예: "강남" → 강남구) 매칭.
3. **동명 지역은 항상 `시군구명(시도)` 배지 병기**하여 각각 별도 행으로 노출한다. 예: 리스트에 `고성군(강원)` `고성군(경남)`이 별개 항목으로 뜬다. 배지는 시각적으로 태그 chip 형태로 구분(회색 배경, 지명과 톤 차이).
4. 리스트 항목 탭 → 입력창에 확정 채움 + `[추측하기]` 버튼 활성화(그 전까지는 비활성 — 자유 입력 텍스트만으로는 제출 불가, 오탭으로 시도 소모 방지).
5. `[추측하기]` 탭이 실제 제출. **선택 + 확인 2탭 구조**로 설계해 오탭 리스크를 낮춘다(시도 6회는 되돌릴 수 없는 자원이므로 즉시제출 UX보다 안전 우선).
6. 이미 추측한 지역은 리스트에서 옅게 비활성 표시(중복 시도 방지, 강제 차단은 아님 — 실수로 같은 지역 다시 골라도 시도만 소모).
7. 탭 타깃 44px 이상, 자동완성 리스트 항목 간격 충분히 확보(엄지 오탭 방지 — WCAG 2.1 AA 타깃 사이즈 기준).

**시도 6회 진행 상태 표시**
- 세로 리스트 6행 고정. 완료된 행: `{번호} {추측 지역명} {거리}km {방향화살표} {근접도}%`. 값은 **자릿수 정렬**(우측 정렬 3칸/2칸 고정폭)로 6행을 스캔하기 쉽게 — 이건 UI디자이너 폰트/그리드 스펙과 연동 필요.
- 진행 전 행: 번호만 옅게 표시(placeholder), 현재 입력 차례인 행은 살짝 강조(좌측 컬러 바 등, UI 담당).
- **정답 시**: 해당 시도 행이 힌트값 대신 "정답! 🎯"로 대체, 이후 행은 숨김(6행 다 채울 필요 없음).
- **실패(6회 소진) 시**: 6번째 행 아래 "정답: {지역명}" 공개 + 실루엣이 라벨과 함께 확정 표시.

**정답/실패 후 공통 — 자정 카운트다운**
- 완료(성공 또는 실패) 상태가 되면 결과 카드 하단에 "다음 문제까지 HH:MM:SS" 실시간 카운트다운 노출.
- 계산은 반드시 `src/lib/dongne/queue.ts`의 KST 고정 UTC+9 산술을 재사용(로컬 타임존/`toLocaleString` 금지 — 해외 접속·기기 시계 스큐 방어).
- 탭 비활성(백그라운드) 시 `setInterval` 카운트다운을 멈추고, 포그라운드 복귀(`visibilitychange`) 시 큐 모듈로 재계산해 리싱크(단순 감산 누적 금지 — 드리프트·자정 넘김 오차 방지).
- 완료 전(진행 중)에는 카운트다운을 굳이 강조하지 않는다(플레이 방해 요소 최소화). 대신 헤더에 "#{gameNo}"만 조용히 표시.

**스트릭 위험 넛지 (웹푸시 없음 — 인앱 배너로만 구현)**
- 조건: 오늘 미플레이 + 기존 streak > 0 + 현재 KST 시각 21:00 이후 → `/dongne` 진입 시 게임 영역 상단에 옅은 배너: "🔥 스트릭 {N}일째! 자정 전에 오늘 문제를 풀어야 이어져요" + 자정까지 남은 시간.
- 세션 단위로만 dismiss(다음 방문 때 조건 충족하면 재노출). 강제 모달 금지 — 배너는 스크롤로 밀려나는 인라인 요소.

### B. 결과 화면 플로우 (인라인 확장, 성공 케이스)

```
정답 확정
  → 해당 시도 행 "정답! 🎯" 전환 (마이크로 애니메이션, 과하지 않게)
  → 결과 카드가 시도 리스트 바로 아래로 슬라이드 확장(auto-scroll로 카드 상단이 뷰포트에 들어오게)
     구성 순서:
       1) "{N}/6 만에 맞혔어요!" 헤드라인
       2) 🔥 스트릭 {N}일 (최고 {M}일) — 항상 노출
       3) 🏅 "오늘 정답이 우리 동네예요!" 뱃지 — 조건부(아래 참조)
       4) [결과 복사하기 📋] 버튼 (1차 수단)
       5) [카톡으로 공유] 버튼 — navigator.share 지원 기기에서만 병행 노출
       6) 다음 문제까지 카운트다운
       7) "어제 정답 알아보기 →" CTA (아카이브 전 회차 링크)
       8) "내 통계 보기 (펼치기)" — 기본 접힘, 탭하면 1~6/실패 히스토그램 확장
```

**공유 텍스트 복사 (카톡 최적화)**
- `[결과 복사하기]` 탭 → `navigator.clipboard.writeText`로 기획서 §3-1 포맷 그대로 복사:
  ```
  동네고수 #123
  3/6 🎯
  1️⃣ 182km ↗ 41%
  2️⃣ 67km ↘ 78%
  3️⃣ 정답!
  🔥 스트릭 5일
  ▶ project-orsrw.vercel.app/dongne
  ```
- 복사 성공 시 버튼 라벨이 일시적으로 "복사됨 ✓"로 바뀌는 **마이크로 피드백**(2초 후 원상복구) + `aria-live="polite"` 스크린리더 안내. 별도 토스트 팝업 대신 버튼 자체 상태 변화로 처리(카톡 인앱은 토스트 위치가 씹히는 경우가 있어 버튼 인라인 피드백이 더 안정적).
- `navigator.share` 지원 기기(iOS Safari 계열 등)에서는 클립보드 버튼 옆에 "카톡으로 공유" 보조 버튼을 병행 노출해 공유 시트를 직접 띄운다. **클립보드가 항상 1차 수단**(방향서 확정 — 인앱 Web Share 불안정 대응).
- 공유 텍스트는 **정답 지역명 비노출**(스포일러 프리) — 온스크린 결과 카드에는 정답 지명이 보이지만, 클립보드로 나가는 텍스트에는 물리적으로 포함시키지 않는다(코드 레벨 분리는 UI/프론트 담당 확인 필요 — 이 UX 스펙에서는 "온스크린 컴포넌트와 공유 텍스트 생성 함수를 분리해 뷰가 곧 공유 텍스트가 되지 않게" 요구사항으로 명시).

**"오늘 정답이 우리 동네" 뱃지 노출 조건**
1. 사전 조건: 사용자가 "내 동네"를 localStorage에 등록한 적 있음(`dongne:myRegionCode`).
   - 미등록 상태로 첫 결과 화면 도달 시, 뱃지 자리 대신 **1회성 유도 문구**를 노출: "내 동네를 등록하면 정답 일치 시 뱃지를 받아요 [내 동네 설정 →]" — 탭하면 동일한 자동완성 컴포넌트가 인라인으로 열려 지역 선택(동명 배지 규칙 동일 적용) → 저장.
   - 등록은 **선택 사항**이며 건너뛰어도 게임 진행에는 영향 없음(비차단).
   - 이후 결과 화면이나 헤더 근처 작은 텍스트 링크("내 동네 변경")로 언제든 재설정 가능.
2. 뱃지 노출 로직: `오늘 정답 행정표준코드 === myRegionCode` 이면 결과 카드에 금박 스타일 뱃지 + 문구 "오늘 정답이 우리 동네였음 ㅋㅋ" 노출. 아니면 뱃지 영역 자체를 렌더하지 않음(빈 상태 방치 금지 — UI 담당과 협의해 카드 높이 튐 최소화).
3. 뱃지는 **정답 확정 이후에만** 계산·노출(진행 중에는 절대 노출 금지 — 스포일러 방지, 미확정 상태에서 뱃지가 뜨면 사실상 정답 유출).

**스트릭 표시**
- 결과 카드 2번째 줄에 항상 "🔥 스트릭 {N}일 (최고 {M}일)" 고정 노출. 오늘이 스트릭 첫날(연속 끊긴 뒤 재시작)이어도 "🔥 스트릭 1일"로 자연스럽게 표시(부끄럽지 않게, 긍정 프레이밍).
- 실패(6/6)로 정답을 못 맞힌 경우 스트릭 처리 정책: **스트릭은 "플레이 여부" 기준**(정답 맞힘 여부 아님)으로 유지 — 6번 다 틀려도 오늘 플레이했으므로 스트릭 유지. 이는 캐주얼 게임 리텐션 원칙(정답률로 스트릭을 끊으면 이탈 유발)에 따른 결정이며, UI/카피에는 "플레이 스트릭"임을 결과 카드 보조 텍스트로 짧게 명시.

**어제의 동네 유도 동선**
- 결과 카드 하단 CTA "어제 정답 알아보기 →"는 `/dongne/archive/{gameNo-1}`로 직행(목록 경유 없이 원탭). 이것이 리텐션+SEO 콘텐츠 층 도달의 핵심 동선이므로 결과 카드에서 시각적 우선순위를 공유 버튼 다음으로 둔다.
- "내 통계 보기"는 그 아래 보조 동선(기본 접힘) — 통계보다 해설 콘텐츠 도달을 우선.

**결과 화면 — 실패(6/6) 케이스 차이점**
- 헤드라인: "6/6 — 아쉬워요!" 톤(비난조 금지).
- 정답 지역명 + 실루엣 확정 표시(실패 시에만 정답 공개).
- 공유 텍스트 포맷은 동일 구조 유지, 마지막 시도 줄이 "정답!" 대신 실제 마지막 힌트값 그대로 노출(스포일러 프리 유지 — 실패해도 공유 텍스트엔 지명 없음).
- "우리 동네" 뱃지 로직은 성공/실패 무관하게 동일 적용(오늘 정답 == 내 동네면 실패했어도 뱃지 노출 — "오늘 정답이 우리 동네였는데 못 맞혔어요 ㅠㅠ" 변형 카피 고려, UI 카피 담당과 협의).

### C. 재방문 플로우 (이미 플레이한 날 재진입)

```
진입 → /dongne 로딩
  → localStorage에서 오늘 gameNo 키로 상태 조회 (`dongne:state:{gameNo}`)
  → 상태 있음
      상태 = "완료(성공)" → 온보딩/입력 단계 전부 스킵, 시도 리스트(과거 기록 그대로) + 결과 카드 바로 렌더
                            (재도전 불가 — 추측 입력 UI 자체를 노출하지 않음, 오늘의 결과 확정임을 명확히)
      상태 = "완료(실패)" → 동일하게 실패 결과 카드 바로 렌더
      상태 = "진행 중(예: 3/6까지 시도, 브라우저 종료 후 재접속)" → 시도 리스트에 기존 1~3행 복원 + 4번째 행부터 입력 재개
                            (온보딩 스트립은 재노출 안 함 — 이미 dismiss됨)
  → 상태 없음 → A(첫 방문 코어 게임 플로우) 그대로 진행
```
- 재방문 판정은 **날짜(gameNo) 단위**. 어제 상태는 오늘 화면에 영향 없음(매일 완전히 새 게임 상태).
- 기기를 바꾸면 로그인이 없으므로 새 기기 = 첫 방문 취급(스코프 내 정상 동작, 계정 시스템은 v2 이연).
- 완료 상태 재방문 시에도 카운트다운·공유·통계 CTA는 동일하게 살아있어야 함(재방문 유저가 스트릭 확인하러 오는 경우가 많음 — 화면이 "이미 다 했다"고 끝나버리면 안 되고 다음 액션(공유, 어제 해설, 내일 예고)을 항상 제시).

## 엣지 케이스

| 케이스 | 설계 대응 |
|---|---|
| **localStorage 사용 불가** (시크릿 모드/차단 설정) | 기능 감지(`try { write/read/remove } catch`)로 사전 판별. 불가 시: (1) 게임은 세션 메모리 상태로만 정상 플레이 가능(진행 중 상태는 유지되나 새로고침하면 초기화 — 이는 허용 가능한 저하), (2) 스트릭/히스토그램/"내 동네"/온보딩 dismiss 등 영속 기능은 조용히 비활성(에러 노출 금지), (3) 결과 화면에 1회성 비차단 안내 문구 "이 브라우저에서는 기록이 저장되지 않아요" 노출(닫기 가능, 매번 재노출은 세션당 1회로 제한), (4) 공유·복사·OG는 storage 무관이라 영향 없음. |
| **자정(KST) 경계에 걸친 플레이** | 진행 중인 라운드는 **세션이 시작된 시점의 gameNo/큐 스냅샷에 고정**한다 — 자정을 넘겨도 이미 열려 있는 화면은 어제 문제를 계속 풀 수 있게 유지(도중에 정답이 바뀌는 불공정 방지). 페이지를 새로고침하거나 재방문하면 그 시점 기준 새 gameNo로 전환되며, 자정 전 완료하지 못한 어제 라운드는 "미완료"로 남고 스트릭에는 포함되지 않는다(단, 스트릭을 강제로 끊지도 않음 — 다음날 정상 플레이하면 스트릭 카운트는 "완료한 날 기준"으로 자연 유지되므로 별도 페널티 로직 불필요). 완료 후 노출되는 자정 카운트다운은 위 A 섹션 규칙대로 `visibilitychange` 시 큐 모듈 재계산으로 드리프트를 방지한다. |
| **아카이브에서 미래/오늘 회차 접근** | `/dongne/archive/[gameNo]`는 `generateStaticParams`가 "오늘 미만"만 생성(SSG 자체가 미래 페이지를 만들지 않음) → 정적으로는 404. 사용자가 URL을 직접 조작해 미래 gameNo로 접근하는 경우를 대비해 **친화적 안내 화면**을 별도로 설계(일반 404 대신): "아직 공개되지 않은 회차예요 · 오늘 문제 풀러 가기 →" — 오늘 게임으로의 복귀 CTA를 반드시 포함(막다른 화면 금지 원칙). |
| **아카이브에서 오늘(gameNo) 회차 접근** | 오늘 회차는 스포 방지 목적으로 아카이브 목록에도, 상세 페이지에도 절대 노출하지 않는다(위 미래 케이스와 동일한 안내 화면으로 처리 — "오늘 문제는 아직 진행 중이에요! 풀러 가기 →" 문구로 톤만 구분). |
| **동명 지역 오추측/오판정 리스크** | UX 레벨 마지막 방어선 — 자동완성 리스트에서 "직접 타이핑 제출" 자체를 허용하지 않고 **리스트 항목 선택만 제출 가능**하게 강제(코드 기반 판정과 UI를 1:1로 묶음). 동명 지역은 항상 배지 병기로 시각 구분. |
| **오늘 실루엣 로딩 실패**(네트워크 지연/오류, 온디맨드 fetch 특성상 발생 가능) | 실루엣 자리에 스켈레톤 → 일정 시간 초과 시 "실루엣을 불러오지 못했어요 [다시 시도]" 인라인 오류 상태. 게임 자체(자동완성 입력)는 실루엣 없이도 진행 가능하게 두어(힌트만으로도 추론 가능) 완전 블로킹은 피한다. |
| **자동완성 결과 0건**(오타/존재하지 않는 지명) | 리스트 영역에 "일치하는 동네가 없어요" 빈 상태 텍스트(에러음/붉은색 남발 금지) — 제출 버튼은 계속 비활성 유지. |
| **네트워크 오프라인 상태에서 재방문** | manifest는 최초 로드 시 캐시되므로 자동완성은 오프라인에서도 동작 가능하나, 오늘 실루엣이 최초 미수신 상태면 위 실루엣 로딩 실패 케이스로 폴백. |

## 인터랙션 노트

- **자동완성 드롭다운 위치**: [F2 확정] 위쪽 오픈 기본, 위 공간<180px면 아래 폴백. [F9 확정] **입력창 non-sticky**(sticky는 헤더만) — 포커스 시 네이티브 scroll-into-view 의존.
- **결과 카드 확장 트랜지션**: 정답/실패 확정 시 시도 리스트 아래로 슬라이드다운(약 200~250ms ease-out), 확장 직후 카드 상단이 뷰포트에 들어오도록 `scrollIntoView({block: 'nearest'})`. 과한 컨페티/사운드 등은 넣지 않는다(카톡 인앱 브라우저 환경, 소음 민원 방지, 성능 예산).
- **복사 버튼 피드백**: 클릭 즉시 라벨 "결과 복사하기 📋" → "복사됨 ✓"로 전환, 2초 후 원복. `aria-live="polite"`로 스크린리더에 "복사되었습니다" 안내. 별도 토스트/스낵바 컴포넌트는 만들지 않는다(카톡 인앱 좌표계 이슈 회피, 버튼 자체 상태 변화로 충분).
- **카운트다운 갱신**: `setInterval` 1초 간격, `document.visibilitychange`에서 숨김 전환 시 인터벌 정지, 복귀 시 `queue.ts` 함수로 재계산 후 재시작(드리프트·자정 오차 방지). 별도 리렌더 최소화를 위해 카운트다운은 독립 컴포넌트로 격리.
- **힌트 행 스크린리더 안내**: 각 시도 제출 결과가 채워질 때 해당 행에 `aria-live="polite"` 영역으로 "1번째 시도, 182킬로미터, 북동쪽, 근접도 41퍼센트" 형태로 낭독. 색(근접도 색상 강조 등은 UI 담당 스펙)만으로 정보 전달하지 않고 항상 숫자 텍스트 병기(WCAG 2.1 AA — 색만으로 의미 전달 금지).
- **탭 타깃**: 모든 인터랙티브 요소(자동완성 항목, 추측하기 버튼, 공유 버튼, CTA 링크) 최소 44×44px, 인접 요소 간 8px 이상 여백.
- **포커스 상태**: 키보드/스크린리더 사용자를 위해 모든 포커스 가능 요소에 명확한 focus-visible 아웃라인 필요(색상·두께는 UI디자이너 토큰 스펙과 연동).
- **온보딩 재노출 금지 로직**: dismiss 시점에 localStorage 플래그 기록. localStorage 불가 환경에서는 매 세션 최초 1회만 노출(반복 노출로 인한 짜증 방지, 완전 억제는 불가하므로 최소화).
- **"내 동네" 설정 컴포넌트 재사용**: 최초 게임 입력의 자동완성 컴포넌트를 그대로 재사용(동일 동명 배지 규칙 적용) — 별도 컴포넌트 신규 제작 없이 프론트 구현 비용 절감.
- **인앱 브라우저 뒤로가기 대응**: `/dongne` 내 결과 인라인 확장은 브라우저 히스토리 스택을 쌓지 않는다(뒤로가기 시 앱 종료/이탈 방지). 아카이브 상세 → 오늘 게임 복귀는 일반 링크 네비게이션(히스토리 정상 사용).

## UI디자이너 전달 사항 (연동 필요 항목)

1. 시도 6행의 거리·방향·근접도 **자릿수 정렬**을 위한 고정폭 그리드/폰트 스펙(모노스페이스 숫자 or tabular-nums).
2. "우리 동네" 뱃지 비주얼(금박 실링 방향서 언급) 및 뱃지 없을 때 카드 레이아웃 높이 처리.
3. 정답/실패 상태의 실루엣 컬러 처리(온스크린은 색 사용 가능 — 단, 공유 텍스트로는 색 정보 절대 유출 금지, 코드 레벨 분리는 프론트팀 확인).
4. 온보딩 스트립 톤(배경색·아이콘) — 모달이 아닌 인라인 스트립임을 강조.
5. 스트릭 위험 넛지 배너 비주얼(경고성이되 공격적이지 않게).
