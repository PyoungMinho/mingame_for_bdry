# 팀메이크 오피스 — UX 설계서 (아이소메트릭 좌표/플로우)

> office-plan.md Phase 1 확정 스코프 기준. 좌표는 전부 "기계가 바로 먹을 수 있는" 정수 격자값으로 확정한다.

---

## 사용자 페르소나

- **PM/사장(사용자)**: 이 오피스를 직접 조작하지 않고 "구경 + 트리거 클릭"만 한다. 목적은 (1) 우리 AI팀이 실제로 존재/작업하는 느낌을 받는 것, (2) 회의 소집 버튼으로 쇼케이스 연출을 보는 것, (3) NPC 클릭으로 팀 구성을 파악하는 것.
- 기술 이해도 높음(개발자), 하지만 이 페이지에서는 **감상자** 역할. 조작은 최소(클릭 1~2회)로 끝나야 함.
- 데스크톱(>=1024px)에서 지나가다 보는 용도 — 세션 길지 않음(30초~2분). 첫 화면에서 "이게 뭔지" 3초 내 이해되어야 함(시뮬레이션 배너 + 방 라벨).

## 사용자 플로우

### 플로우 A — 랜딩 & 관찰 (기본 진입)
```
1. /office 진입
   → 로딩(오프스크린 캔버스 프리렌더) 최대 500ms 목표
2. 스테이지 페이드인, 상단 SIMULATION 배너 즉시 노출
3. 6룸 + 회의실 아이소 배경 정적 렌더 확인
4. NPC 17명이 각자 room의 home 좌표에서 idle 애니(2프레임 bob) 시작
5. 낮/밤 60초 루프가 자동 재생 시작 (초기 진입 시점 = "낮" 단계로 고정 시작)
6. 좌하단 컨트롤 패널(회의 소집 버튼 + 낮밤 스크러버) 상시 노출
→ 여기서 사용자는 아무 것도 안 눌러도 "살아있는 오피스"를 감상 가능 (핵심 액션 없이도 완결)
```

### 플로우 B — NPC 클릭 → 프로필 확인 (핵심 액션 1)
```
1. 사용자가 임의 NPC 스프라이트를 클릭
2. 히트박스 판정 (자세 규격은 아래 "인터랙션 플로우" 참조)
3. 클릭된 NPC 기준 좌상단 방향으로 ID카드 팝업 소환 (등장 tilt 트랜지션 150ms)
4. 팝업 내용: 팀컬러 헤더 스트립 + 이름 + 역할(모노) + 현재 상태(idle/walking/working/meeting) + 프로시저럴 미니 아바타
5. 팝업 바깥 클릭 or ESC or 다른 NPC 클릭 시 즉시 닫힘/전환
→ 목표: 클릭 1번으로 3초 내 "이 사람 누구고 뭐하는 중인지" 파악
```

### 플로우 C — 회의 소집 연출 (핵심 액션 2, 킬러 기능)
```
1. 좌하단 컨트롤 패널 [회의 소집 ▸] 버튼 클릭
2. (버튼이 팀별 드롭다운을 겸함 — 아래 "회의 소집 UX 결정" 참조) 팀 선택
3. 버튼 상태 → "소집 중..." (비활성화, 중복 클릭 방지)
4. 해당 팀 NPC들이 순차적으로 room → 복도 → meeting-room 경로로 walking 애니와 함께 이동 (팀별 스태거 0.3s)
5. 전원 도착 시 meetingSeats 좌표에 착석, state="meeting"으로 idle 전환 (팀컬러 아지랑이 없이 담백하게)
6. 12초 유지("회의 중" 표시는 회의실 상단 라벨로 대체 — 말풍선은 Phase 2 컷)
7. 자동으로 각자 origin room의 home 좌표로 walking 복귀
8. 버튼 원상태로 재활성화
→ 사용자는 이 전체(약 20~30초)를 관찰만 하면 됨. 재생 중 다른 팀 소집 버튼은 비활성화(동시 회의 금지 — 좌석 수 제약)
```

### 에러/빈 상태 설계 (먼저 설계)
- **로딩 실패/캔버스 미지원**: "이 브라우저는 캔버스를 지원하지 않습니다" 중앙 카드 + 새로고침 버튼. 그레인 오버레이 없이 담백한 잉크 배경.
- **모바일(<1024px) 진입**: 전체화면 안내 카드 — "팀메이크 오피스는 데스크톱 전용입니다 (최소 1024px)" + 최소 아이소 배지 아이콘 1개(장식용, 무거운 애셋 금지). 스테이지 자체를 마운트하지 않음(성능 보호).
- **회의 소집 재클릭(이미 진행 중)**: 버튼 disabled + 툴팁 "이미 회의가 진행 중입니다" (모노 텍스트, 앰버 보더 없이 뮤트 톤).
- **NPC 클릭 판정 실패(빈 공간 클릭)**: 아무 반응 없음(무소음 실패 — 에러 토스트 금지). 열려있던 팝업은 닫힘.
- **초기 프레임 드랍(성능 하한 미달 기기)**: 자동 감지 없이 Phase 1은 그대로 진행하되, 컨트롤 패널에 "속도 토글"로 60초 루프를 90초로 늘리는 완화 옵션만 제공(자동 저하 로직은 Phase 2).

## 정보 구조 (IA)

```
/office (단일 라우트, 화면 전환 없음 — SPA 캔버스 1장)
│
├── [DOM 레이어] 상단 시뮬레이션 배너 (고정, z-top)
├── [DOM 레이어] 코너 타이틀 락업 "팀메이크 오피스 / TEAM MAKES · OFFICE" (고정)
├── [Canvas 레이어] 아이소 스테이지
│   ├── 배경 6룸 + 회의실 + 복도
│   ├── NPC 17명 (idle/walking/meeting)
│   └── 조명 오버레이(4단계 틴트 + 램프 글로우, additive)
├── [DOM 레이어] 좌하단 컨트롤 패널 (고정, z-mid)
│   ├── 회의 소집 트리거 (팀 선택 겸용)
│   ├── 낮/밤 스크러버 (현재 단계 표시 + 수동 스크럽 가능)
│   └── 속도 토글 (1x / 느리게)
└── [DOM 레이어] NPC 프로필 팝업 (조건부 렌더, 클릭 시에만, z-top)
```

- 네비게이션 없음(단일 화면 완결형 — 뎁스 0). "3탭 이내 목적 달성" 원칙 자동 충족(클릭 1회로 팝업, 1회로 회의 소집).
- 딥링크/상태 URL 파라미터는 Phase 1 범위 밖(공유 URL로 특정 조명 단계 고정 등은 Phase 2 검토).

## 아이소 투영 & 스테이지 상수

```jsonc
{
  "tile": { "tileW": 64, "tileH": 32 },   // 2:1 다이아몬드
  "origin": { "screenX": 720, "screenY": 160 }, // 캔버스 내 grid(0,0)의 화면 좌표(px). 스테이지 중앙 상단 기준
  "canvas": { "width": 1440, "height": 900 },   // 권장 스테이지 캔버스 (데스크톱 1024~1440 반응형 스케일, 이 값을 디자인 기준 해상도로 사용)
  "gridBounds": { "cols": 22, "rows": 16 },     // 전체 그리드 범위(0..21, 0..15)
  "projection": {
    "cartToIso": "screenX = origin.screenX + (gx - gy) * (tileW/2); screenY = origin.screenY + (gx + gy) * (tileH/2)",
    "isoToCart": "gx = ((screenX - origin.screenX)/(tileW/2) + (screenY - origin.screenY)/(tileH/2)) / 2; gy = ((screenY - origin.screenY)/(tileH/2) - (screenX - origin.screenX)/(tileW/2)) / 2"
  },
  "zSort": "renderOrder = gx + gy (오브젝트/NPC의 anchor 타일 기준), 동률이면 objectType 우선순위(바닥<벽<데스크<NPC) 후 삽입순"
}
```

- 오브젝트 압출 높이는 렌더 규격(디자인팀 팔레트 지침에 따름): 윗면(밝음) / 좌측면(중간) / 우측면(어두움) 3면 프리즘, 벽 높이는 타일 1.5배(48px) 기준.
- 접지 그림자: NPC/데스크 anchor 타일 중심에 반투명(alpha 0.18) 타원, 반경 tileW*0.35 x tileH*0.35, 블러 없이 소프트 페더 4px.

## 회의실 좌석 좌표 (meetingSeats)

회의실 origin (8,7), size 6x4. 최대 동시 착석 인원 = 팀 최대 규모(기획팀 5명) 기준 6석 배치, 직사각 테이블 둘레.

```jsonc
{
  "meetingTable": { "gx": 10, "gy": 8, "w": 2, "h": 2 },  // 테이블 오브젝트 anchor
  "meetingSeats": [
    { "seatId": 0, "gx": 9,  "gy": 7 },
    { "seatId": 1, "gx": 10, "gy": 7 },
    { "seatId": 2, "gx": 11, "gy": 7 },
    { "seatId": 3, "gx": 9,  "gy": 10 },
    { "seatId": 4, "gx": 10, "gy": 10 },
    { "seatId": 5, "gx": 11, "gy": 10 }
  ],
  "seatAssignPolicy": "소집 팀 인원 수만큼 seatId 0부터 순번 배정. isLead 멤버 우선 seatId 0(테이블 상단 정중앙, 가장 눈에 띄는 자리) 배정."
}
```

- 6인 초과 팀 없음(최대 기획팀 5명)이라 6석으로 전 시나리오 커버.
- 착석 시 NPC는 테이블을 바라보는 방향으로 스프라이트 flip(상단 좌석=남향, 하단 좌석=북향).

## NPC 루틴 이동 웨이포인트 규칙

### 기본 idle 루틴 (회의 소집 없을 때)
```
state = "idle" 고정, home 좌표에서 2프레임 세로 bob 반복.
간헐적(30~45초 랜덤 간격) 마이크로 워크: home ↔ 반경 1~2타일 내 보조 지점(예: 프린터/책장 오브젝트) 왕복 1회.
  → 이건 Phase 1 "stretch"로 분류(스코프 문서 세부 데코와 동급 우선순위), 없어도 무방. 있으면 생동감 ↑.
```

### waypoint 경로 규칙 (room → meeting-room)
```
경로 = [home] → [방 door 좌표] → [세로 복도 gx=7 상 교차점] → [회의실 인접 door] → [배정 seatId 좌표]
이동은 격자 셀 단위 스텝(매 스텝 = 1 타일), 팀 내 스태거 0.3s로 순차 출발(스코프의 "1초 틱 이동결정 + rAF 보간" 구조 그대로 적용).
경로 계산은 사전 정의 waypoint 배열(런타임 pathfinding 없음 — 6룸 고정 배치라 A* 불필요, 하드코딩 경로표로 충분).
```

**팀별 사전 정의 경로 (room door → 회의실 door)**:

| team | room door | 중간 교차점(복도) | 회의실 진입 door |
|---|---|---|---|
| planning | (6,3) | (7,3) → (7,6) | (8,8) |
| design | (8,3) | (7,3)→(7,6) 또는 직진 (8,4)→(8,6) | (10,7) |
| frontend | (14,3) | (13,3)→(7,3 방향 아님, 우측 진입) (13,6) | (13,8) |
| backend | (6,12) | (7,12)→(7,11) | (8,8) |
| qa | (8,13) | (8,12)→(8,11) | (8,8) 또는 (11,7) |
| deploy | (15,12) | (14,12)→(13,11) | (13,8) |

> 실제 구현 시 위 표를 `waypoints[team] = Point[]` 상수 배열로 하드코딩. 복도는 격자 gx=7 세로대 + gy=6/gy=11 가로대로 구성되어 있어 각 팀 room door에서 최단 L자 경로로 연결 가능.

## 회의 소집 안무 타임라인 (초 단위)

전체 길이 목표: 약 28~34초 (팀 규모에 따라 변동, 기획팀 5명 기준 최댓값 예시).

```
t=0.0s   사용자가 [회의 소집 ▸] → 팀 선택 확정
t=0.0s   버튼 상태 "소집 중..." 전환, 재클릭 방지
t=0.3s   1번째 NPC(isLead) walking 애니 시작, home → door로 이동
t=0.6s   2번째 NPC 출발 (스태거 0.3s)
t=0.9s   3번째 NPC 출발
t=1.2s   4번째 NPC 출발 (팀 5인일 때)
t=1.5s   5번째 NPC 출발
  ※ 이동 소요시간은 경로 길이(타일 수) x 0.35s/타일 가정. 평균 경로 6~8타일 → 약 2.1~2.8s
t≈3.5~5s NPC들이 순차적으로 meeting-room door 도착 → 배정 seatId로 착석, state="meeting" 전환
t=5s     전원 착석 완료(가장 늦은 NPC 기준), 회의실 상단에 얇은 라벨 "○○팀 회의 중" 페이드인(모노, 팀컬러 언더라인)
t=5~17s  "회의 중" 유지 (12초 고정 — 스코프의 10~20초 범위 중간값)
t=17s    복귀 시작 신호, 라벨 페이드아웃
t=17.3s~ 각 NPC 역순 스태거 0.3s로 seatId → door → home 역경로 walking
t≈20~22s 전원 home 도착, state="idle" 복귀
t=22s    버튼 원상태 재활성화("회의 소집 ▸")
```

- 스태거·이동속도 상수는 코드 튜닝 여지를 남기되, **골격(스태거 0.3s, 유지 12s, 이동 0.35s/타일)** 은 이 문서 기준값으로 고정.
- 회의 중 다른 팀의 회의 소집 버튼은 자동 비활성화(동시 회의는 회의실 좌석/애니메이션 충돌 방지 위해 Phase 1에서 금지 — 큐잉도 없음, 단순히 "이미 진행 중" 안내만).

## 인터랙션 플로우 (클릭/컨트롤)

### NPC 클릭 히트박스
```
히트박스 = NPC anchor 타일의 화면 투영 좌표 기준 사각형
  width: 28px, height: 40px (발밑 anchor에서 위로 확장 — 머리/몸통 전체 커버)
  anchorOffset: 히트박스 하단 중심 = NPC 발밑 그림자 중심과 일치
우선순위: 겹치는 히트박스 있을 시 renderOrder(zSort) 값이 큰(화면상 더 앞) NPC 우선 판정
```

### 클릭 → 프로필 팝업
```
트리거: 히트박스 내 mousedown/click (모바일 터치는 Phase 1 미지원 — 데스크톱 전용이므로 click만)
팝업 위치: 클릭된 NPC 화면 좌표 기준, 기본은 우상단 방향 오프셋(+16px, -140px)
  단, 팝업이 캔버스 우측 경계(width-320px) 넘어가면 좌상단 방향으로 자동 반전(+16px→-320px)
  단, 팝업이 캔버스 상단 경계(0px) 넘어가면 NPC 하단 방향으로 반전
팝업 크기: 288 x 160px 고정
등장 트랜지션: opacity 0→1 + scale 0.94→1 + rotate -2deg→0deg, 150ms ease-out (카드가 "톡" 놓이는 느낌)
닫힘 트리거: 팝업 바깥 클릭 / ESC 키 / 다른 NPC 클릭(즉시 이전 팝업 닫고 새 팝업, 트랜지션 재생) / 60초 무동작 시 자동 닫힘(선택)
포커스 관리: 팝업 열릴 때 role="dialog" aria-live="polite"로 스크린리더에 이름/역할/상태 안내 (접근성, 아래 상세)
```

### 컨트롤 패널 (좌하단 고정)
```
위치: position fixed, left: 24px, bottom: 24px, z-index: 상 (팝업보다는 아래, 배너보다는 위)
구성 (세로 스택, 인셋 패널 카드 1개 안에 3영역):
  1. [회의 소집 ▸] — 클릭 시 팀 6개 드롭업 메뉴 노출(기획/디자인/프론트/백엔드/QA/배포), 팀 선택 즉시 안무 시작
     - 진행 중 상태: 버튼 라벨 "소집 중... (○○팀)" + disabled 스타일(앰버 텍스트 → 뮤트 텍스트)
  2. 낮/밤 스크러버 — 가로 슬라이더 + 현재 단계 모노 라벨(예: "07:40 · 낮"), 4단계 구간 눈금 표시
     - 자동 재생 중에도 드래그로 수동 스크럽 가능 → 스크럽 즉시 해당 단계로 틴트 전환, 이후 자동 루프는 그 지점부터 재개
  3. 속도 토글 — "1x / 0.6x"(느리게) 2단 세그먼트 버튼, active 상태 앰버 채움
전체 패널: 인셋 그림자(box-shadow inset) + 소프트 드롭섀도, 배경은 플럼-브라운 반투명, active 요소만 앰버 강조(anti-generic — 균일 강조 금지, 딱 하나만 튐)
```

## 데스크톱 레이아웃 (스테이지 + 크롬, >=1024px)

```
┌─────────────────────────────────────────────────────────────┐
│ ● SIMULATION — 실제 에이전트는 로컬에서 동작        (얇은 스트립, 전체폭, 상단 고정) │
├─────────────────────────────────────────────────────────────┤
│ 팀메이크 오피스                                                │  ← 코너 타이틀 락업
│ TEAM MAKES · OFFICE                        (좌상단, 스테이지 위 오버레이) │
│                                                                 │
│                    [ 아이소 스테이지 캔버스 ]                    │
│                    1440 x 900 (비율 유지 스케일)                 │
│                    - 뷰포트 1024~1439px: 캔버스 0.72x~1x 스케일   │
│                    - 뷰포트 1440px 이상: 1x, 캔버스 중앙 정렬     │
│                    - 세로 여백은 딥 잉크 bg + 그레인 + 비네트     │
│                                                                 │
│ ┌──────────────┐                                              │
│ │ 컨트롤 패널    │                            (좌하단 고정)       │
│ │ 회의소집/낮밤/속도│                                              │
│ └──────────────┘                                              │
└─────────────────────────────────────────────────────────────┘
```

- **뷰포트 대응**: 1024px 미만은 스테이지 자체 미마운트, 안내 카드만(위 "에러/빈 상태" 참조). 1024~1439px는 캔버스를 `transform: scale()`로 축소(내부 좌표계는 1440x900 고정 유지 — 좌표 재계산 없이 CSS 스케일만).
- **z-index 레이어 순서(아래→위)**: 페이지 bg(그레인) < 캔버스 스테이지 < 좌하단 컨트롤 패널 < 코너 타이틀/상단 배너 < NPC 프로필 팝업(최상단).

### 접근성 노트 (WCAG 2.1 AA)

1. **색 대비**: 텍스트 #f2ede3(웜 오프화이트) on 배경 #15161d/#3a2f3a 조합은 AA 이상 확보(명도 대비 계산상 7:1대). 뮤트 텍스트(#9a9488)는 큰 텍스트/보조 정보에만 사용(AA large 기준).
2. **캔버스 콘텐츠의 대체 텍스트**: Canvas 렌더링은 스크린리더가 읽지 못하므로, 캔버스 위/아래 시각적으로 숨긴(`sr-only`) 요약 텍스트 제공 — "팀메이크 오피스: 17명의 AI 에이전트가 기획/디자인/프론트엔드/백엔드/QA/배포 6개 팀에서 근무 중인 아이소메트릭 시뮬레이션 화면입니다."
3. **키보드 접근성**: 컨트롤 패널의 회의 소집 버튼/드롭업/스크러버/속도 토글은 전부 실제 `<button>`/`<input type=range>` DOM 요소로 구현(캔버스 안에 그리지 않음) → 자연히 키보드 tab/enter/화살표 조작 가능. NPC 클릭은 마우스 전용이나, 필수 정보(팀 구성)는 대체 텍스트로 이미 제공되므로 키보드 전용 사용자도 핵심 정보 접근 가능.
4. **포커스 표시**: 모든 인터랙티브 DOM 요소에 명확한 focus-visible 아웃라인(앰버 2px) — 다크 배경에서 특히 중요.
5. **애니메이션 민감도**: `prefers-reduced-motion: reduce` 감지 시 — NPC walking 애니를 즉시 이동(텔레포트)으로 대체, 낮/밤 틴트 전환은 크로스페이드 대신 즉시 전환, 팝업 등장 트랜지션(scale/rotate) 생략하고 opacity만.
6. **팝업 dialog 시맨틱**: `role="dialog"` `aria-labelledby`(이름) `aria-describedby`(역할+상태), 닫힘 후 포커스는 트리거였던 NPC의 논리적 포커스 앵커(숨김 버튼)로 복귀.
7. **배너 깜빡임**: SIMULATION 배너의 앰버 점 깜빡임은 `prefers-reduced-motion` 시 고정 점등으로 전환(발작 유발 애니메이션 가이드라인 준수 — 저빈도 단순 페이드라 기본적으로 안전하나 완화 옵션 제공).
