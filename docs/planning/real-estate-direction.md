## 0. 이 방향서의 목적 — 기획서 대비 확정/변경 사항

기획서(`real-estate-plan.md`)의 **전략·타깃·콘셉트는 전부 유지**한다. 이 방향서는 그 위에서 팀장 3인의 기술·실현 검토를 받아 **"실제로 어떻게 만들지"를 확정**하는 문서다. 핵심 변경은 3가지다.

| # | 항목 | 기획서(v1.0) | 방향서(확정) | 변경 이유 |
|---|---|---|---|---|
| 1 | **신뢰 스코어 배점** | 사진40/등기20/EXIF15/신고15/거래완료10 (외부 의존 30) | 사진35/EXIF·생활흔적20/신고·커뮤니티20/소유자검증15/거래정황10 (외부 의존 ~10) | 백엔드 검증: 등기·거래완료의 1:1 자동 매칭이 API상 불가·고비용 |
| 2 | **기술 스택** | 미확정(보류 항목) | **Supabase BaaS 코어 + 전용 워커 레이어 하이브리드** | 프론트(BaaS 속도) vs 백엔드(엔진 제어력) 충돌을 경계선으로 해소 |
| 3 | **통합 일정** | M0 ~6주(단일) | M0를 **8주**로 현실화(디자인 선행 3주 → 개발 병렬), PoC 2주 병행 | 디자인→프론트 의존 + 외부 PoC 리스크 반영 |

> 한마디로: **콘셉트는 그대로, 스코어는 자체 신호 중심으로 재설계, 스택은 하이브리드, 일정은 M0 8주로 현실화.**

## 1. 확정 기술 스택 (결정 2)

### 결단: **Supabase를 코어 BaaS로, 신뢰/사진/외부연계는 전용 워커 레이어로 분리한 하이브리드.**

프론트팀장(Supabase 단독 재활용)과 백엔드팀장(NestJS 별도 서버)의 충돌을, **"무엇을 BaaS에 맡기고 무엇을 직접 통제할지"의 경계선**으로 해소한다. MVP에서 별도 풀 백엔드 서버(NestJS)를 띄우는 것은 일정·운영 부담이 과해 **채택하지 않는다.** 다만 백엔드팀장이 옳게 지적한 영역(스코어 엔진의 결정성, 잡큐 제어, 원본 사진 비공개, 외부 ACL 격리)은 **BaaS에 녹이지 않고 별도 워커로 독립**시킨다.

#### 경계선 — 무엇을 어디서

| 책임 영역 | 담당 | 근거 |
|---|---|---|
| 인증(본인인증 게이트 연동·세션) | **Supabase Auth** + PASS/통신사 어댑터 | 자체 구현 금지(백엔드팀장). Auth는 BaaS로 충분 |
| 관계형 데이터·지리쿼리 | **Supabase Postgres + PostGIS 확장** | PostGIS는 Supabase에서 확장으로 가용 → 별도 DB 불필요 |
| 실시간 스코어 반영 UI | **Supabase Realtime 구독** | 프론트팀장 안 채택(폴링/구독, 낙관적 UI 지양) |
| 파일 저장(원본 비공개/공개본 분리) | **Supabase Storage**(비공개 버킷 + 서명 URL) | 원본은 비공개 버킷, 블러 통과본만 공개 버킷 |
| **신뢰 스코어 엔진** | **전용 워커**(Edge Functions + DB 트리거/잡 테이블) | 결정성·재계산 제어가 핵심. 프론트 정렬·낙관적 UI 금지 |
| **사진 파이프라인**(EXIF·AI 블러·생활흔적) | **전용 워커**(서버 측 처리, 큐 기반) | 클라 EXIF는 위변조 가능 → 반드시 서버. 프론트팀장 ★ |
| **외부 연계**(등기/공공API) | **ACL 어댑터 모듈**(워커 내, 인터페이스로 격리) | 백엔드팀장 ACL 안. 외부 실패=pending 처리 |
| 무한스크롤·신뢰순 정렬 | **서버 사전계산 + 인덱스 컬럼**, 커서 페이지네이션 | 프론트 정렬 금지(프론트팀장 ★) |

#### 프론트엔드 스택 (프론트팀장 안 그대로 채택)
- **Next.js 14.2.3 (App Router) + React 18 + TypeScript strict + Tailwind 3.4 + React Query 5 + Zustand 4 + Radix UI + Supabase JS** — 레포 기존 자산(oreum 잔재) 재활용.
- **신규 도입은 지도 SDK뿐** — 단, 지도뷰는 디자인팀장 판단대로 **P1**이므로 M0에서는 도입 보류, M1 진입 시 카카오맵/네이버맵 중 택1(PoC 항목).
- 프론트 컴포넌트: `TrustScoreBadge`, `ListingCard`, `PhotoUploader`, `ReportSheet`, `VerificationStepper`.
- 프론트 페이지: `/`(검색), `/listings/[id]`(상세 RSC), `/verify`(업로드 클라), `/report`.

#### 워커 레이어 구현 방식
- **1순위: Supabase Edge Functions**(Deno) + **잡 테이블(Postgres) 기반 큐** + **pg_cron/DB 트리거**로 시작. 별도 인프라 0으로 MVP 속도 확보.
- **확장 신호 시 분리**: 사진 AI 블러·생활흔적 추론이 Edge Function 실행 한도(시간/메모리)를 넘기면, 그 단계만 **별도 경량 Node 워커**(컨테이너)로 떼어낸다. — 이 분기 판단은 M0 W2 PoC 결과로 확정.

> **결정 요지**: 풀 NestJS 서버는 안 띄운다(속도). 그러나 엔진·사진·외부연계는 BaaS 자동화에 의존하지 않고 **우리가 코드로 통제하는 워커**로 둔다(제어력). 양 팀장 우려를 동시에 충족.

## 2. 수정된 신뢰 스코어 설계 (결정 1)

### 원칙
백엔드팀장 검증을 전면 수용한다: **외부 의존 배점(기존 등기20+거래완료10=30점)을 ~10점대로 축소**하고, **우리가 직접 통제 가능한 자체 신호로 재배분**한다. 외부 검증 실패는 **0점이 아니라 `pending`**(미검증 ≠ 위반)으로 처리하며, 항목별 점수와 `verifiedAt`를 **분리 저장**해 부분 신선도를 표기한다.

### 새 배점표 (100점 만점)

| 항목 | 배점 | 검증 방법 | 통제 | 신선도(verifiedAt) |
|---|---|---|---|---|
| **실거주 세입자 사진** | **35** | 본인인증 업로드 + 앱내/직촬 + 공개 게시 통과 | 자체 | 업로드 시각 |
| **EXIF·생활흔적 정합** | **20** | 서버 EXIF 대조(GPS·타임스탬프) + AI 생활흔적 스코어 | 자체 | 처리 완료 시각 |
| **커뮤니티 신뢰**(신고 이력) | **20** | 신고 0건 만점, 누적 시 차감 + 검수 큐 진입 | 자체 | 최근 신고/검수 시각 |
| **소유자·등록자 검증** | **15** | 중개사 자격 확인(기본) + **집주인 동의 시 등기 1회 조회 가산** | 혼합 | 검증 시각 |
| **거래 정황 플래그**(간접) | **10** | "동일 동·기간 거래 급증" 정황 플래그(공공 통계 기반) | 외부(간접) | 배치 갱신 시각 |
| **합계** | **100** | | | |

> **외부 직접 의존은 사실상 0**, 외부 간접 의존(거래 정황 10점 + 소유자 검증의 등기 가산분)만 남긴다. 외부 데이터가 비거나 지연돼도 스코어 산출이 멈추지 않는다.

### 항목 설계 상세
- **세입자 사진(35)**: 기존 40 → 35. 여전히 최대 단일 항목이자 차별점의 핵심. 단, EXIF·생활흔적을 별도 항목(20)으로 떼어 "사진을 올렸다"와 "그 사진이 진짜 그 집에서 찍혔다"를 분리 채점 → 어뷰징(남의 집 사진 붙이기) 방어가 점수 구조에 내재.
- **EXIF·생활흔적(20)**: 기존 15 → 20. 자체 통제 가능하고 어뷰징 방어의 핵심이라 비중 강화.
- **커뮤니티 신뢰(20)**: 기존 15 → 20. 사용자 신고가 외부 API보다 신뢰도 높고 즉시성 있는 자체 신호.
- **소유자·등록자 검증(15)**: 기존 등기 20 → 15로 축소 + 의미 재정의. **전 매물 자동 등기 검증은 폐기**(건당 ~700원+동의 비현실). 기본은 등록자(중개사) 자격 확인, **집주인이 동의한 경우에 한해 1회 등기 조회**로 가산.
- **거래 정황 플래그(10)**: 기존 "거래완료 자동 강등" 폐기. 공공 통계는 법정동+계약년월 단위라 개별 매물 1:1 매칭 불가 → **"동일 동·기간 거래 급증"을 간접 정황 플래그**로 재정의. 자동 강등이 아니라 **검수 큐 우선순위 신호 + 소폭 감점**으로만 사용.

### 컷오프 (배지 등급)
- **🟢 Gold (실거주 인증)**: **80점 이상**
- **🟡 Silver (현장 인증)**: **55 ~ 79점**
- **⚪ Unverified (미인증)**: **55점 미만**

> 기획서 컷오프(80/60~79/미만)에서 Silver 하한을 **60 → 55로 하향**한다. 근거: 외부 자동 강등(거래완료 10점)이 사라지고 등기 가산이 선택적이 되면서, 정직한 공실 매물(중개사 앱내 촬영 + EXIF 정합 + 신고 0)이 60점 문턱을 못 넘어 Unverified로 떨어지는 사각이 생긴다. 콜드스타트 공급 보호(기획서 DP-1·DP-3 정신)를 위해 Silver 진입을 5점 완화. **수치는 M1 실데이터로 재튜닝**(미해결 항목).

### 배지·UI 규칙 (디자인팀장 안 채택)
- **색 + 형태 이중 코드화(색맹 대응 필수)**: Gold=방패+체크, Silver=카메라핀, Unverified=점선 회색.
- **점수 표기**: 숫자 + 미니 도넛. 상세 화면에선 **5개 항목 막대 분해로 "왜 82점인지" 투명 공개** — 직방·다방과의 차별 체감 핵심.
- **사람 언어 우선**: 숫자보다 "실거주 세입자가 찍음" 같은 자연어 라벨을 먼저 노출.
- **Unverified는 숨기지 않되 앰버 경고 톤**(노출은 열고 신뢰로 줄 세운다는 기획 원칙과 일치).

## 3. 외부 데이터 연계 방침

백엔드팀장 검증 결과를 정책으로 확정한다.

### 핵심 원칙
1. **외부는 격리(ACL)**: 등기·공공API는 **어댑터(ACL) 모듈** 뒤에 둔다. 외부 스키마·장애·정책 변경이 우리 도메인/스코어 로직을 오염시키지 않게 인터페이스로 분리.
2. **실패 = pending, 0점 아님**: 외부 조회 실패·지연·미동의는 해당 항목을 `pending` 상태로 두고 **미검증 ≠ 위반**으로 표기. 스코어는 나머지 자체 신호로 산출.
3. **하이브리드 재계산**: 사용자 액션(사진 업로드·신고)은 **이벤트 즉시 재계산**, 외부 의존 신호는 **비동기 잡큐 + 배치**로 갱신. 항목별 `verifiedAt` 분리 저장으로 부분 신선도 노출.

### 데이터별 처리
| 데이터 | 현실 | 우리 처리 |
|---|---|---|
| **등기부등본** | IROS 공식 API는 통계 한정. 실발급은 CODEF/하이픈 등 민간 스크래핑 건당 과금(~700원+) | **전 매물 자동 검증 폐기.** "집주인 명시 동의 시 1회 조회"로 한정 → 소유자 검증(15) 가산분 |
| **실거래/전월세/임대차** | 공공데이터포털 무료·즉시 가용이나 법정동+계약년월 통계 단위, 30일+ 지연, 1:1 매칭 불가 | **"거래완료 자동 강등" 폐기.** "동일 동·기간 거래 급증" 간접 정황 플래그(10점)로 재정의 + 검수 큐 신호 |
| **본인인증** | 자체 구현 시 보안·법 리스크 | **PASS/통신사(나이스·드림시큐리티) 어댑터**. 자체 구현 금지 |

## 4. 화면 구조 / IA 요약 (디자인팀장 반영)

핵심 원칙: **MVP는 "진짜인지"가 본질이므로 리스트 우선, 지도뷰는 P1로 미룬다.**

### P0 (M0에 반드시)
1. **검색**(`/`): 신뢰순 정렬 기본(서버 사전계산), 커서 무한스크롤.
2. **매물 상세**(`/listings/[id]`, RSC): 신뢰 스코어 패널 + **5개 항목 막대 분해** + 신고 진입.
3. **세입자 사진 인증 업로드**(`/verify`): 본인인증 게이트 → 직촬 유도 → 진행 스텝바.
4. **본인인증·전자동의 게이트**: 업로드 전 PASS 인증 + 버전드 전자동의.

### P1 (M1 이후)
- 신고 플로우 고도화, **이사확인 인증서 발급함**, 중개사 매물관리(앱내 카메라), **지도뷰**(+지도 SDK 도입).

### 업로드 UX 난점 대응 (디자인팀장 안)
- **4단계 이탈**: 진행 스텝바 + PASS 1탭.
- **EXIF 탈락 좌절**: **앱내 카메라 직촬 유도** + 사유 명시(외부 업로드는 EXIF 신뢰 불가).
- **블러 신뢰**: before/after 프리뷰 + 수동 블러 보정.
- **즉시 피드백**: 업로드 직후 "+점수/상단노출" 즉시 표시.

## 5. 사진 처리 파이프라인 (프론트/백엔드 책임 분리)

프론트팀장 ★(클라 EXIF 금지) + 백엔드팀장 파이프라인 안을 결합해 확정한다.

```
[프론트] /verify
  PASS 본인인증 → 전자동의(버전드 로그) → 앱내 카메라 직촬/선택
  → 업로드 진행률 표시 → 처리상태 폴링/Realtime 구독 (※ 낙관적 UI 지양)
        │  (원본 파일만 전송, 클라에서 EXIF·블러 처리 안 함)
        ▼
[워커/서버 파이프라인]
  ① 비공개 버킷 저장(원본, 절대 미공개)
  ② 큐 적재 → ③ EXIF 대조(GPS·타임스탬프 정합)
  → ④ AI 블러 게이트(얼굴·송장·서류·소지품) — 통과 필수
  → ⑤ 생활흔적 스코어링
  → ⑥ 블러 통과본만 공개 버킷 게시 + 스코어 항목 갱신(이벤트 재계산)
        ▼
[프론트] 등급 판정 결과 수신 → 배지·점수·"+상단노출" 피드백
```

### 책임 분리 (명확화)
| 단계 | 담당 | 금지사항 |
|---|---|---|
| EXIF 추출·검증 | **서버/워커** | 클라 EXIF 추출 금지(위변조·우회 가능, 스코어 근거 불가) |
| AI 블러링 | **서버/워커** | 블러 미통과본 공개 금지 |
| 생활흔적 스코어 | **서버/워커** | — |
| 원본 저장 | **비공개 버킷** | 원본 공개 절대 금지, 공개본은 블러 통과본만 |
| 진행률·결과 표시 | **프론트** | 낙관적 UI 금지(서버 확정 결과만 표시) |

### 동의·테이크다운 (백엔드팀장 안)
- **버전드 전자동의 로그**: IP·타임스탬프·약관버전을 **불변(immutable) 기록**.
- **notice-and-takedown**: 신고 시 **즉시 비공개** 전환.

## 6. 통합 마일스톤 (결정 3)

3개 팀 일정(디자인 ~3주, 프론트 M0 6주, 백엔드 코어 6주+PoC 2주)을 **디자인→프론트 선행 의존**을 반영해 통합한다. **M0를 8주로 현실화**한다(단일 6주는 디자인 선행분이 누락된 수치).

### M0 — 기반 구축 (8주, 디자인 선행 + 개발 병렬)

| 주차 | 디자인팀 | 프론트팀 | 백엔드/워커팀 |
|---|---|---|---|
| **W1** | 디자인 토큰 + **배지 시스템** 확정 | 프로젝트 셋업(기존 스택 검증) | 스키마(항목별 점수·verifiedAt 분리) + Auth/Storage 셋업 |
| **W2** | 검색/상세 화면 디자인 | 셋업 마무리 + 컴포넌트 골격 | **스코어 엔진 코어** + **외부 ACL PoC 시작** |
| **W3** | 업로드 플로우 디자인 + **핸드오프** | 검색·상세 구현(디자인 수령분) | 스코어 엔진 + **사진 파이프라인(EXIF/블러) 착수** |
| **W4** | (지원·QA 협업) | 검색·상세 마무리 | 사진 파이프라인 + **ACL PoC 결과 보고** |
| **W5** | — | 업로드 UI(PhotoUploader/Stepper) | 블러 게이트·생활흔적 + Realtime 연동 |
| **W6** | — | 업로드 마무리 + 폴링/구독 연결 | 스코어 이벤트 재계산 + 잡큐 안정화 |
| **W7** | — | 신고 플로우 + 통합 | 신고→스코어 차감→검수 큐 |
| **W8** | — | 폴리싱 + E2E | 통합·부하·신선도 표기 검증 |

> **의존 규칙**: 디자인 **토큰·배지(W1)**가 프론트 컴포넌트의 선행 조건 → W1에 최우선 산출. 검색/상세 디자인(W2)이 프론트 구현(W3)을 막지 않도록 1주 리드. 외부 ACL은 **PoC(W2~W4)**로 분리해 본 일정 리스크에서 격리.

### M1~M4 (기획서 계승, 기간 확정)
| 단계 | 기간 | 목표 | 완료 기준 |
|---|---|---|---|
| **M1. 공급 채우기** | ~8주 | 중개사 무료 온보딩 + 세입자 사진 캠페인 + **지도뷰(P1)** + 컷오프 재튜닝 | 권역별 인증 50건 / 전체 200건 |
| **M2. 수요 점화** | ~6주 | 소비자 마케팅(밀도 게이트 통과 후) | WAU 확보, 문의 발생 |
| **M3. 루프 검증** | ~8주 | 세입자→소비자→재인증 선순환 측정 | 인증 매물 문의 전환율 유의차 확인 |
| **M4. 수익화 실험** | M3 이후 | SaaS 유료 전환 + 리드 광고 베타 | 유료 전환 의향/실제 결제 검증 |

## 8. 다음 단계 — 디자인 착수 준비사항

디자인팀이 M0 W1에 바로 착수할 수 있도록 다음을 입력으로 전달한다.

1. **확정 스코어 배점표·컷오프**(2장): 배지는 5개 항목 막대 분해를 전제로 디자인. 등급 컷오프 80/55, 색+형태 이중 코드.
2. **배지 시스템이 최우선 산출물**(W1): Gold=방패+체크 / Silver=카메라핀 / Unverified=점선 회색 + 앰버 톤. 미니 도넛 + 자연어 라벨("실거주 세입자가 찍음").
3. **P0 화면 4종**: 검색(신뢰순)/상세(스코어 패널)/업로드(스텝바)/인증·동의 게이트. **지도뷰는 P1이므로 M0 디자인 범위에서 제외.**
4. **업로드 플로우 제약 반영**: 앱내 카메라 직촬 우선, before/after 블러 프리뷰, 진행 스텝바, 업로드 직후 점수 피드백.
5. **상태 디자인 필수**: `pending`(외부 미검증 — 위반 아님), 처리중(폴링/구독), 신고됨(테이크다운) 등 비동기 상태의 시각 언어.
6. **기존 스택 제약 인지**: 프론트는 Radix UI + Tailwind 3.4 → 디자인 토큰을 Tailwind 설정과 정합되게 설계.

> 이 방향서 승인 후 → **"디자인 시작"** 트리거로 디자인팀 3인(UX/UI → 팀장) 가동.
