## 0. 전원 판정 — GO (조건부)

| 팀장 | 판정 | 핵심 조건/수정 |
|---|---|---|
| **디자인팀장** | GO (조건부) | G2용 '공평성 시각화' 프로토타입을 디자인팀 1순위로 먼저 입증 |
| **프론트팀장** | GO | W1에 카톡 인앱브라우저 호환 스파이크 선행 + 실시간 투표는 폴링부터 |
| **백엔드팀장** | GO (수정 1건) | G3 시뮬레이션에 'OSRM 1차필터 호출 절감율'을 변수로 명시 |

**종합 판정: 3팀장 전원 GO. 세 가지 조건/수정을 정식 게이트 및 일정에 통합하여 진행한다.**

전원이 동의한 합의점은 다음과 같다.

1. **MVP 2축 고정** — (1)다자간 중간지점 추천(공평지수) (2)카톡 링크 무가입 그룹 투표+과반 자동확정. 예약·정산은 딥링크 패스스루만.
2. **G2(추천 품질 우위)가 프로젝트의 생사** — 디자인은 "시각화로", 백엔드는 "알고리즘으로" G2를 함께 입증한다. 미통과 시 프로젝트 중단.
3. **'공평성=편차(격차) 최소화'가 카카오맵이 못하는 차별점** — 평균이 아닌 '막내 1시간 vs 부장 10분'의 불공평을 균등화하고, 그것을 수치(백엔드)와 가로막대(디자인)로 동시에 보여준다.
4. **비용 통제가 G3의 본질** — 진짜 비용은 시간매트릭스 N×K 호출. OSRM 사전필터·격자캐싱·무가입투표 0원으로 흑자 경로를 만든다.
5. **과설계 금지** — 서버리스 + 기존 `src/app/api` 확장. 웹소켓·네이티브·자체 정산엔진은 MVP에서 제외.

## 1. 확정 기술 스택 & 아키텍처 (한눈에)

### 한 줄 스택
> **Next.js(App Router, 투표·공유 SSR/RSC) + 카카오맵 SDK + ODsay/카카오로컬/자체 OSRM + Postgres+Redis 서버리스**

### 프론트엔드
| 영역 | 선택 | 근거 |
|---|---|---|
| 프레임워크 | Next.js App Router | 투표·공유 페이지 SSR/RSC (OG 메타가 바이럴 좌우 → SSR 필수) |
| 지도 | 카카오맵 JS SDK (client component) | 다자 핀 시각화 |
| 카톡 공유/투표 | Kakao.Share + 무가입 웹뷰 (SSR + 서버액션) | 로그인 0, 즉시 투표화면 |
| 상태/데이터 | Tailwind + SWR/Zustand | 경량 |
| 실시간 집계 | **폴링 + 서버액션** (웹소켓 아님) | MVP 단순화, 슬립 리스크 회피 |

### 백엔드
| 영역 | 선택 | 근거 |
|---|---|---|
| 런타임 | 서버리스 (Next API Routes) | 과설계 금지, 기존 src/app/api 확장 |
| DB / 캐시 | Postgres + Redis | 관계형 + 격자캐싱 |
| 대중교통 경로 | ODsay (주력) | 환승·소요시간 |
| 지오코딩/POI | 카카오 로컬 | 주소→좌표, 장소 후보 |
| 거리 사전필터 | **자체 호스팅 OSRM (컨테이너 1)** | 0원, K 후보 사전 가지치기 |
| 데이터 모델 | meet / participant / place_candidate / vote / share_link | 토큰=nanoid(16), 만료 7일 |

### 공평성 알고리즘 (G2 승부처 — 백엔드 확정안)
centroid(무게중심)는 **시드일 뿐**(강 건너·환승지옥 결함). 3단계로 확정:
1. **후보역 탐색** — centroid 반경 내 지하철·환승역 K=15~20 추출
2. **시간 매트릭스** — N명 × K후보 대중교통 소요시간 + 환승횟수 산출
3. **공평지수** — `score = α·평균 + β·최대 + γ·표준편차` (환승 가중). **min(max)+분산 페널티**로 '최악 멤버'와 '격차'를 함께 최소화.

### 보안
- `voter_token` = 링크토큰 + 브라우저 fingerprint + UNIQUE 제약 + IP 레이트리밋 (중복·어뷰징 차단)
- 위치 PII 암호화 저장 + **확정 후 좌표 파기**
- 카톡 토큰은 서버 시크릿, 클라이언트 비노출. Kakao.init 가드.

## 4. 검증 게이트 업데이트 (3팀장 의견 반영)

> 기획서의 G1·G2·G3를 유지하되, 팀장 검토로 **운영 책임·측정 변수**를 추가 확정.

### 🚦 G1 — 수익 경로 실재성 (기획 주관, 변경 없음)
- 통과: 예약 가맹점 직계약 10곳+ 구두 합의 **또는** 법인 3곳+ "유료로 쓰겠다" 확인 중 하나.
- 미통과: 예약 커미션 완전 제외, B2B 게이트로 전환.

### 🚦 G2 — 추천 품질 우위 (**디자인+백엔드 공동, 필수**)
- 통과: 블라인드 테스트 20명에서 **모이라 추천 ≥ 카카오맵 수동검색 60%+ 선호** (특히 공평성·이동시간).
- **[디자인 조건 반영]** Phase 0에서 **'공평성 시각화' 프로토타입을 디자인팀 1순위로 먼저 입증** — 편차(격차)를 가로막대로 보여주는 것이 카카오맵 대비 우위 포인트.
- **[백엔드 직감 반영]** G2 통과의 **7할은 알고리즘 품질**. 3단계 공평지수가 '최악 멤버·격차'를 수치로 균등화함을 입증.
- **미통과 시: 프로젝트 자체 NO-GO.** (차별점이 없으면 존재 이유 없음)

### 🚦 G3 — 단위경제 흑자 시뮬레이션 (**백엔드 수정 반영**)
- 통과: MAU 1만/5만/10만 구간별 (API+인프라) vs 수익 시뮬에서 5만 내 흑자 경로 존재.
- **[백엔드 수정 반영 — 필수]** 시뮬레이션에 **'OSRM 1차필터 호출 절감율'을 독립 변수로 명시**. (캐싱보다 큰 레버 — K 20→6 사전필터 시 시간매트릭스 호출 ~70%↓). 격자캐싱 히트율(50%+)과 분리하여 두 레버를 각각 모델링.
- 미통과: OSRM 절감율·캐싱율 재설계 또는 B2B 단가로 수익 측 보강.

### 🚦 (신규) W1 카톡 스파이크 — 프론트 선행 조건 (Phase 0 내)
- **[프론트 조건 반영]** v0.1 착수 전 W1에 카톡 인앱브라우저 호환을 실기기로 검증 통과해야 무가입 투표 본개발 진입. (구형 WebView·iOS 외부스킴·도메인/JS키 등록)
- 실시간 투표는 웹소켓 금지, **폴링+서버액션으로 시작**.

> **개발 GO 조건**: G2 통과(필수) + G3 통과 + W1 스파이크 통과. G1은 6주차 수익 경로 분기를 가른다.

### 최종 방향 확정 문구

> ## ⚖️ **확정: 3팀장 전원 GO — 조건부 풀개발 진입**
>
> - **GO 하는 것**: Next.js 웹앱 + "중간지점(공평지수) + 카톡 무가입 투표 자동확정" MVP 2축. 예약·정산은 딥링크만.

> - **유보·6주차 결정**: B2C 바이럴 확장 vs B2B 회식 SaaS — v0.1 지표+G1로 정식 분기.
>
> **한 줄 결단**: *"공평성은 '편차'로 증명한다 — 디자인이 막대로 보여주고 백엔드가 수치로 증명하는 그 한 화면이 G2이고, G2가 모이라의 생사다."*
