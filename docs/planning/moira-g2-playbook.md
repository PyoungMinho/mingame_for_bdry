# 모이라 G2 블라인드 테스트 운영 키트 (Playbook)

> **이 문서는 "설계서"가 아니라 "실행 키트"입니다.** 사장님(PM)이 이 문서 하나만 들고 20명에게 직접 돌릴 수 있게 만든 운영 매뉴얼입니다.
> G2의 정의·근거·합격선 자체는 이미 확정돼 있습니다 → [`midpoint-meetup-direction.md`](midpoint-meetup-direction.md) §G2, [`moira-routemap-techreview.md`](moira-routemap-techreview.md) §4-3, [`moira-vision-revision.md`](moira-vision-revision.md). 이 키트는 **그걸 실제로 돌리는 절차**만 담습니다.

---

## 0. 한 줄 요약 — G2가 뭐고, 왜 지금인가

**G2 = "우리 추천이 카카오맵보다 나은가"를 사람 20명에게 블라인드로 묻는 시험.**
**60% 이상(20명 중 12명+)이 모이라를 고르면 PASS → 라이브 개발 착수. 미만이면 STOP.** 이게 모이라의 생사 게이트입니다.

지금 데모(`/moira` 5화면)는 **G2를 측정하기 위한 도구**입니다. 0원으로 돌고(목 데이터·SVG 폴백), 따로 서버·DB·API 키가 없어도 됩니다. 그래서 라이브 개발에 돈·시간 쓰기 **전에** 이 시험부터 통과시키는 게 순서입니다.

---

## 1. 합격/불합격 기준 (외우세요)

| 항목 | 기준 |
|---|---|
| 표본 | **친구 아닌 20명** (지인 편향 = 결과 무효) |
| 핵심 질문 | "둘 중 어느 쪽으로 약속을 잡고 싶나요?" (택1) |
| **PASS** | 모이라 선택 **≥ 12명 (60%)** |
| **FAIL** | 모이라 선택 ≤ 11명 → **프로젝트 STOP** |
| 기권/동점 처리 | "모르겠다"는 **모이라 표 아님**으로 집계(보수적). 분모는 20 고정 |

> ⚠️ 합격선을 시험 **시작 전에** 못 박으세요. 끝나고 "11명도 거의 됐으니까…" 식으로 옮기면 시험 자체가 무의미해집니다.

## 2. 데모 준비 (사장님이 시험 전에 1번)

### 2-A. 데모 띄우기 (둘 중 택1)

**① 로컬 (가장 간단, 0원)**
```bash
npm run dev
# http://localhost:3000/moira 부터 시작
```

**② 외부 배포 (참가자에게 링크만 보낼 때 — 권장)**
- Vercel 등에 배포 후 받은 URL 사용. 참가자가 본인 폰으로 들어오게 하면 "모바일 우선" 의도대로 측정됩니다.

### 2-B. 지도 타일 — 기본은 그대로 둬도 됨

- **그대로(0원):** 카카오 키가 없으면 경로지도가 **SVG 폴백**으로 뜹니다. 시험은 이대로 충분합니다(우리가 묻는 건 "지도 예쁨"이 아니라 "공평함이 보이나").
- **실타일 쓰고 싶으면(선택):** 사장님이 [카카오 디벨로퍼스](https://developers.kakao.com)에서 JS 앱키를 직접 발급 → 프로젝트 루트에 `.env.local` 만들고 아래 한 줄:
  ```
  NEXT_PUBLIC_KAKAO_MAP_KEY=발급받은_JS_키
  ```
  → `npm run dev` 재시작. (키 발급은 제가 못 합니다 — 사장님 계정 인증이 필요한 외부 작업입니다.)

### 2-C. 시험에 쓸 화면 URL

| 단계 | URL | 용도 |
|---|---|---|
| 결과(추천) | `/moira/result?view=ready` | `?view=ready`로 로딩 연출 스킵 → 바로 결과. 참가자 대기 0 |
| **경로지도 ★** | `/moira/routes` | G2의 핵심 화면. 4명 경로·시간격차·장소교체 |
| 투표 | `/moira/vote` | 무가입 투표 체감(선택 측정) |

> 데모 시나리오는 **서울 4인 약속**으로 고정돼 있습니다: 민호(강남 역삼)·서연(노원 상계)·지훈(사당)·예린(홍대). 추천역 **을지로3가(2·3호선)**. 이 시나리오를 카카오맵에도 **똑같이** 넣어 비교합니다.

---

## 3. 비교 대상(카카오맵) 세팅

같은 4인 시나리오로 카카오맵에서 "약속 장소 정하기"를 재현합니다. 가장 공정한 방법:
1. 카카오맵 앱/웹에서 **을지로3가**를 검색해 보여주거나,
2. 4명 출발지(역삼·상계·사당·홍대) 각각 → 을지로3가 길찾기를 **따로따로** 보여줍니다.

→ 핵심은 **"카카오맵은 한 명씩 따로 봐야 하고, 4명을 한 화면에서 공평하게 비교해 주지는 않는다"**는 점이 자연스럽게 드러나게 하는 것. (편들지 말고, 있는 그대로 보여주세요.)

---

## 4. 블라인드 A/B 프로토콜

1. **순서 랜덤화:** 참가자 번호 **홀수 → 카카오맵 먼저**, **짝수 → 모이라 먼저**. (먼저 본 쪽에 쏠리는 편향 상쇄)
2. **블라인드:** 진행자는 **어느 쪽이 "우리가 만든 것"인지 말하지 않습니다.** "두 가지 방법" 이라고만 부릅니다(A안/B안).
3. **동일 조건:** 둘 다 같은 4인 시나리오. 한쪽만 더 오래 설명하지 않기.
4. 각각 **40초~1분** 둘러보게 한 뒤 질문지로.

## 5. 진행자 스크립트 (그대로 읽으세요)

> "친구 **4명**이랑 만나서 밥 먹기로 했는데, 다들 사는 데가 달라요 — 강남, 노원, 사당, 홍대. **어디서 보면 다 같이 공평할지** 정해야 해요. 두 가지 방법(A·B)을 보여드릴게요. 정답은 없고, **어느 쪽이 더 정하기 편한지**만 봐주세요."

**(A안·B안 차례로 보여주며)**

> "여기서 보실 건 **선이 예쁘냐가 아니라, 누가 얼마나 가고 — 그게 얼마나 공평한지**예요." ← *(프레이밍 가드: 곡선 경로선은 데모용 근사치라 모양이 아니라 '시간 격차'를 보게 유도)*

**모이라(경로지도)에서 꼭 하게 할 2가지:**
- **S1 · 4명 한눈에:** 멤버 칩(민호·서연·지훈·예린)을 켜고 끄며 **"누가 제일 멀고 가까운지" 한 화면에서** 보게 하기.
- **S2 · 장소 바꿔보기:** 장소를 A→B→C로 바꿔보게 하기. **공평 점수가 82 → 74 → 68로 실시간으로 바뀜** → "장소를 바꾸면 누가 손해인지 바로 보인다"를 체감.

**(둘 다 본 뒤)**

> "자, 둘 중에 **실제로 친구들이랑 약속 잡을 때 쓰고 싶은 쪽**은 어디예요?"

## 6. 측정 질문 (채점지/구글폼 그대로 복붙)

**Q1. (핵심·필수)** 두 방법 중 실제로 약속 잡을 때 쓰고 싶은 쪽은?
- ☐ A안   ☐ B안   ☐ 모르겠다
  *(진행자만 아는 매핑표로 나중에 모이라/카카오 환산)*

**Q2. (이유·정성)** 그 이유를 한 문장으로:
`__________________________________________`

**Q3. (공평성 체감)** "누가 더 멀고, 얼마나 공평한지"가 잘 보였나요? *(모이라 기준)*
- 1(전혀) – 2 – 3 – 4 – 5(아주 잘)

**Q4. (재사용 의향)** 다음에 여러 명 약속 잡을 때 이런 걸 쓸 의향?
- 1(전혀) – 2 – 3 – 4 – 5(꼭)

> Q1만으로 PASS/FAIL을 가립니다. Q2~Q4는 **왜 졌는지/이겼는지**를 알아 다음 결정을 돕는 보조 지표입니다.

## 8. 판정 후 — 다음 행동 (분기)

### ✅ PASS (모이라 ≥ 12/20)
→ **라이브 개발 게이트 착수 승인.** 이 순서로(이미 리뷰에서 식별된 블로커):
1. 🔴 **`src/lib/moira/server/token.ts` fail-open → fail-closed 전환** (운영에서 시크릿 미설정 시 거부)
2. **마이그레이션 작성** `003_moira_rls.sql` · `004_moira_cron.sql`(좌표 파기)
3. **프론트 데이터페칭 레이어 + 투표 폴링** 연결, **ODsay 실폴리라인**으로 경로선 교체
> 단, G3(유닛이코노믹스)·W1(카톡 인앱 스파이크)도 라이브 GO 조건에 함께 걸려 있음 — direction 문서 참조.

### ❌ FAIL (모이라 ≤ 11/20)
→ **STOP.** 라이브 개발 착수 금지. Q2/Q3로 원인을 가르세요:
- **추천 품질 문제**(장소·역이 별로다, 카카오가 더 정확) → 알고리즘이 G2의 7할. 백엔드 공평지수 재설계 없이는 재도전 무의미.
- **시각화 전달 문제**(공평한 건 알겠는데 안 와닿는다) → 디자인 재작업 후 재시험 가능.
- 둘 다 약하면 → 제품 가설 자체 재검토(피벗/종료).

## 9. 운영 함정 체크리스트 (시험 망치는 실수들)

- [ ] **친구·가족 넣지 않기.** 1명만 섞여도 표본이 오염됩니다. "약속 자주 잡는데 이 데모 처음 보는 사람"이 이상적.
- [ ] **진행자가 모이라 편들지 않기.** "이게 우리가 만든 거예요"는 절대 금지. A/B로만.
- [ ] **순서 랜덤화 지키기**(홀짝 규칙). 다 모이라부터 보여주면 결과 못 씀.
- [ ] **같은 시나리오 고정.** 사람마다 다른 출발지 넣지 말 것 — 4인(강남·노원·사당·홍대)으로 통일.
- [ ] **"선 모양 보지 말고 시간 격차 보라"** 안내 빠뜨리지 않기(곡선은 데모 근사치).
- [ ] **결과 미리 노출 금지.** 대기 중인 다음 참가자에게 앞사람 화면/대화 안 보이게.
- [ ] **표본 ≥ 20 채우기.** 15명에서 멈추면 60% 라인이 흔들립니다.

## 10. 빠른 시작 (3줄)

1. `npm run dev` → 폰/브라우저로 `/moira/result?view=ready` 와 `/moira/routes` 확인.
2. 6~7장 §5 스크립트 + §7 채점지 인쇄(또는 구글폼). 카카오맵 비교화면(§3) 준비.
3. 친구 아닌 20명에게 §4 블라인드로 → §7 집계 → §8 분기.

> **요약:** 데모는 시험 칠 준비가 끝났습니다(목 데이터·SVG 폴백으로 0원 구동, tsc·테스트·브라우저 회귀 0). 남은 건 **사람 20명** — 그건 사장님이 모으셔야 하는 부분이라 이 키트로 핸드오프합니다.
