> **컨셉 한 줄**: 브라우저에서 여는 아이소메트릭 2D 가상 오피스 — 우리 AI 팀 17명이 NPC로 방을 돌아다니고, 트리거 한 번에 회의실로 모여드는 "살아있는 회사".

배포: 루트 `src/app/office/` → `git push` → `https://project-orsrw.vercel.app/office` (Canvas 직접 구현, 게임 엔진 의존성 없음).

## 결론 요약

창의팀의 "회의 소집 연출 원톱" 방향과 비평팀의 "엔진 3종으로 스코프 확정" 방향을 **결합**한다. Phase 1은 (A) **엔진 3종**(아이소 배경 6룸 · NPC 17명 루틴 이동 · 클릭 이름표)을 반드시 완성하는 뼈대로 삼고, 그 위에 창의팀의 **킬러 연출 1개(회의 소집)** 를 유일한 "쇼케이스 기능"으로 얹는다. 낮/밤 3단계 조명은 비용 대비 효과가 좋아 유지, 그 외(미니맵·카메라줌·말풍선·occupied 진행바·감정 애니·모바일)는 전부 컷/이월한다. 에셋은 **자체 제작이 아니라 기존 무료 아이소 타일셋 커스터마이즈**로 디자인팀 역할을 재정의해 최대 리스크(에셋 파이프라인)를 원천 축소한다. status.json은 Phase 1에서 **읽지 않되 데이터 스키마와 렌더러 인터페이스를 미리 확정**해, Phase 2 전환을 조건부 분기(기술부채) 대신 "동일 스키마를 먹는 두 데이터소스 어댑터"로 설계한다.

## 채택한 아이디어 (창의팀)

| 채택 | 이유 |
|---|---|
| **회의 소집 연출** (Phase 1 원톱 킬러) | "오피스가 살아있다"를 단일 장면으로 가장 압축적으로 증명. Vercel 링크 하나로 "우리 AI팀이 실제로 회의하러 걸어간다"는 스토리텔링 완성. 비평팀도 별도 반대 없음 |
| **NPC 17명 + 걷기/idle 애니** | 엔진의 기본 골격. 17명 규모는 유지하되 워크사이클 2~3프레임으로 최소화 |
| **클릭 시 프로필 팝업 (이름+역할+상태 1줄)** | 구현 비용 낮고 "탐험 재미" 가성비 최고. 비평팀도 "클릭 이름표"로 동일 지지 |
| **낮/밤 3단계 조명 (60초 타임랩스 루프)** | 양측 모두 "저비용 고효과" 합의. 실시간 타이머 대신 데모용 압축 루프 |
| **status.json 스키마 선반영** | 창의팀 "데이터 스키마만 공유·로직 분리"안 + 비평팀 "조건부 분기 회피"안이 정확히 합치 |

## 반영한 리스크 대응 (비평팀)

1. **스코프 크립(근본 원인)** → Phase 1 커트라인을 "엔진 3종 + 킬러연출 1 + 낮밤"으로 못박고 나머지는 명시적 컷 라벨. 이 하나로 에셋·성능·일정 리스크가 연쇄 완화됨.
2. **에셋 파이프라인 부재** → 디자인팀 역할을 "제작"에서 **"기존 무료 아이소 타일셋 조합·커스터마이즈"**로 재정의. Figma는 색/배치만 조정. NPC는 4방향 × 2~3프레임 단순 스프라이트.
3. **Z-order 깊이 정렬 과소평가** → 방을 **겹치지 않는 격자 셀로 분리**해 NPC 발밑 y좌표 단일 기준 정렬로 충분하게 설계. 방 경계를 렌더 레이어로 분리해 이동 중 정렬 사이클(순환 참조) 원천 차단. (토폴로지 정렬의 O(n²)·cycle 문제를 애초에 안 만나는 격자 제약 방식)
4. **17명 성능 미검증** → 시뮬레이션(이동 결정)은 ~1초 틱으로 배칭, 화면 부드러움은 `requestAnimationFrame` **보간(interpolation)**으로 분리. (AI Town은 엔진 60틱 + DB write 스텝 배칭 + history 재생 보간 구조 — 우리는 규모가 작으니 이동결정 1틱 + 렌더 보간으로 단순화). 오프스크린 캐싱 전제. NPC 1→5→10→17 단계 프레임 측정을 QA 게이트로.
5. **Vercel 데모 vs 로컬 라이브 기대치 불일치** → 페이지 상단 **"시뮬레이션 데모" 배너** 필수 포함 + Phase 2 로드맵 본 문서에 명시. 에이전트는 로컬 Claude Code 세션에서만 도므로 Vercel /office는 필연적 스크립트 시뮬레이션임을 UI로 고지.

## Phase 1 확정 스코프 (구현할 것)

1. **아이소메트릭 오피스 배경 — 6룸** (다이아몬드 타일 격자, 벽/바닥/책상/오브젝트, 짧은 그림자로 입체감만). 세부 데코는 stretch.
2. **NPC 17명 렌더링 + 걷기 애니** (4방향 × 2~3프레임). 각 NPC는 `{ agentId, name, role, room, state, x, y }` 로컬 상태 객체.
3. **NPC 자리 idle 애니** (방 성격별 반복 모션 1종: 펜 까딱 / 마우스 흔들 / 타이핑, 2~3프레임 루프).
4. **NPC 루틴 이동** (waypoint 경로 스크립트). 격자 셀 기반 이동으로 z-sort 단순 유지.
5. **회의 소집 연출** (데모 버튼 → 해당 팀 NPC가 방에서 나와 회의실 테이블로 걸어와 모임 → "회의 중" idle → 10~20초 후 복귀).
6. **클릭 프로필 팝업** (NPC 히트박스 클릭 → 이름/역할/상태 1줄).
7. **낮/밤 3단계 조명** (주간/저녁/야간 색 오버레이, 60초 데모 루프).
8. **status.json 스키마 + 렌더러 데이터 어댑터 인터페이스 선반영** (파일 안 읽음, 형태만 확정).
9. **"시뮬레이션 데모" 상단 배너** (기대치 관리).
10. **데스크톱 전용** (min-width 1024px, 모바일은 안내 화면).

## Phase 2 스코프 (다음)

- **로컬 라이브 연동**: `status.json` 실제 폴링 → 실제 팀 에이전트 작업 상태를 NPC state/room/activity에 반영 ('기획 회의' 트리거 → 기획실 5명 '작업중').
- **데이터소스 어댑터 분기**: 파일 유무로 자동 분기 (있으면 라이브 어댑터, 없으면 데모 스크립트 어댑터) — 렌더러는 동일.
- **실시간 말풍선** (activity/message를 방 위 말풍선으로), **회의실 occupied 진행바**.
- **NPC 하루 풀 루틴 스케줄** (9~22시 다단계 이동).
- **모바일/반응형** 지원 검토.

## status.json 브릿지 스키마 초안 (Phase 2용, Phase 1에 구조 반영)

Phase 1은 **읽지 않지만** NPC 상태 객체와 렌더러 입력을 이 형태로 맞춘다. Phase 2는 이 파일을 폴링하는 어댑터만 추가.

```jsonc
{
  "version": 1,
  "generatedAt": "2026-07-06T12:00:00+09:00",  // ISO8601
  "mode": "live",                               // "live" | "demo" (렌더러 배너 결정)
  "activeWorkflow": "planning" | null,          // 현재 구동 중 워크플로우 (트리거명)
  "agents": [
    {
      "agentId": "planner-lead",                // 안정적 고유 키 (한글명과 분리)
      "name": "기획팀장",
      "team": "planning",
      "role": "최종 결정권자",
      "room": "planning-room",                  // 아래 배치표의 roomId
      "state": "working",                        // "idle" | "walking" | "working" | "meeting"
      "activity": "최종 결론 종합 중",            // 짧은 현재 작업 (말풍선/팝업용)
      "message": "양측 의견 검토 완료",           // 선택: 최근 한마디 (Phase2 말풍선)
      "targetRoom": "meeting-room",             // 이동 목적지 (walking일 때), 없으면 null
      "progress": 0.6,                           // 0~1, occupied 진행바용 (선택)
      "updatedAt": "2026-07-06T12:00:00+09:00"
    }
  ],
  "rooms": [
    { "roomId": "meeting-room", "occupied": true, "occupantIds": ["planner-lead"] }
  ]
}
```

- **설계 원칙**: `agentId`는 UI 문자열(한글명)과 분리한 안정 키. `state`/`room`/`targetRoom`만 있으면 렌더러가 이동·모임 연출을 전부 구동할 수 있게 함(연출 로직은 데이터소스 무관).
- **어댑터 계약**: `getOfficeState(): Promise<OfficeState>` 하나로 통일. Phase 1 = `demoScriptAdapter`(스크립트 생성), Phase 2 = `statusFileAdapter`(status.json 폴링). 렌더러는 어느 어댑터인지 모른다 → 조건부 분기 기술부채 회피.
- **파일 위치(Phase 2)**: 레포 루트 `status.json` (로컬 CLI 세션이 write, dev 서버가 read). Vercel엔 파일 없음 → 자동으로 demo 어댑터.

## 방 ↔ 에이전트 배치표

| roomId | 방 이름 | 팀 | 인원 | 에이전트 (agentId) |
|---|---|---|---|---|
| `planning-room` | 기획실 | 기획팀 | 5 | 기획팀장(planner-lead) · 창의기획자(creative-lead) · 창의팀원(creative-member) · 비평기획자(critic-lead) · 비평팀원(critic-member) |
| `design-studio` | 디자인 스튜디오 | 디자인팀 | 3 | 디자인팀장(design-lead) · UX디자이너(ux-designer) · UI디자이너(ui-designer) |
| `dev-room` | 개발실 | 프론트엔드팀 | 3 | 프론트팀장(fe-lead) · 컴포넌트개발자(fe-component) · 페이지개발자(fe-page) |
| `server-room` | 서버실 | 백엔드팀 | 3 | 백엔드팀장(be-lead) · API설계자(be-api) · DB설계자(be-db) |
| `qa-lab` | QA랩 | QA팀 | 2 | QA설계자(qa-designer) · QA실행자(qa-runner) |
| `deploy-room` | 배포실/옥상 서버 | 배포팀 | 1 | 배포엔지니어(deploy-eng) |

## 실행 계획 (다음 단계)

1. **배경 먼저**: 무료 아이소 타일셋 선정 → Figma에서 색/배치 조정 → Canvas 좌표 매핑으로 6룸 정적 배경 렌더 (에셋 파이프라인 실검증이 첫 관문).
2. **NPC 데이터 모델 + 어댑터 계약** 확정 (`getOfficeState()` + 위 스키마) → `demoScriptAdapter` 구현.
3. **격자 셀 기반 이동 + y좌표 z-sort** → 걷기/idle 애니 연결.
4. **회의 소집 연출** 스크립트 (팀별 목적지=meeting-room → 이동 → 모임 → 복귀).
5. **클릭 프로필 팝업** + **낮/밤 60초 오버레이 루프** + **시뮬레이션 배너**.
6. **성능 게이트**: NPC 1→5→10→17 단계 프레임 측정 (Chrome Perf/Lighthouse), 오프스크린 캐싱 확인 후 배포.

## 보류 / 추가 검토 사항 (오픈 이슈)

1. **에셋 라이선스**: 선정할 무료 타일셋의 상업/배포 라이선스(재배포·수정 허용) 확인 필요 — 데모 공개 전 필수. (디자인팀 확인)
2. **성능 상한 미정**: 17명 + 6룸 실측 프레임을 아직 안 잡음 — 개발 초기에 WebGL 전환 필요 여부 조기 판단.
3. **회의 소집 트리거 UX**: Phase 1 데모 버튼을 어디에(팀별 버튼 vs 단일 "회의 시작") 배치할지 디자인팀 결정 대기.
4. **status.json write 주체(Phase 2)**: 로컬 CLI 세션에서 실제로 이 파일을 어느 시점에 쓸지(각 에이전트 호출 훅) — Phase 2 착수 전 별도 기술 스파이크 필요.
5. **이스터에그 대사(Phase 1.5)**: 말풍선 인프라 없이 팝업에만 한 줄 넣는 축소판을 Phase 1에 끼울지 여부 — 저비용이라 재검토 여지.
